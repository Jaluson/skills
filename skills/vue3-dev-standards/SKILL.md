# Vue 3 标准化开发规范 — AI 自动化执行流水线

本 skill 不是"建议文档"，而是 AI 执行 Vue 3 开发任务时**必须严格遵循的执行流水线**。
每个阶段都有明确的输入、输出和质量门禁（Quality Gate），只有通过当前门禁才能进入下一阶段。
任何阶段发现阻塞问题时，必须回退到对应阶段修正，禁止带病推进。

### 三条不可违反的铁律

**铁律一：不明确就提问，禁止猜测。**
需求中任何模糊、有歧义、或可以有多种理解的表述，都必须向用户提问澄清。
宁可多问一句，也不能凭猜测写出一堆不符合用户预期的代码。
判断标准：如果对一个需求点你能想到两种及以上的理解方式，就必须提问。

**铁律二：分析完毕必须展示理解，等待用户确认后才动手。**
每完成一个分析阶段（P0/P1/P2），必须向用户主动展示你对需求的理解、
你的发现和你的计划。用户说"可以""没问题""继续"后才能进入下一阶段。
禁止跳过确认直接开始编码。

**当用户要求跳过确认时**：用户说"别问了直接做""快速搞一下""跳过吧"等，
按以下规则处理：

| 确认点 | 用户说跳过时 | 原因 |
|--------|------------|------|
| P0 需求澄清 | **可合并**：将 P0/P1/P2 合并为一次展示 | 需求已足够清晰时，三次确认确实冗余 |
| P1 变更范围 | **可合并**：与 P2 合并展示 | 信息重叠度高 |
| P2 方案设计 | **不可跳过**：必须展示方案并等待确认 | 这是编码前的最后防线，必须确保方向正确 |

合并后的最小确认格式：
```
【需求 + 方案确认（合并展示）】
需求理解：（一句话）
方案概述：（两三句话）
关键设计决策：（列出非直觉的选择）
变更文件：（列出）

确认后开始编码。
```
即使合并，P2 的方案确认点也**不可省略**。

**铁律三：动手前检查，有没有相关 skill 可以辅助。**
在进入编码之前，检查当前环境是否有可用的相关 skill（如代码审查、测试、
部署等）。如果有，在合适的阶段主动使用它们，而不是全部手动处理。
特别注意：如果存在 `dev-task-precheck` skill，编码任务应优先通过它作为入口，
由它完成前置分析后再交接给本 skill，而非直接触发本 skill。
如果本 skill 是被 `dev-task-precheck` 交接触发的，接收交接包中的预扫描结果
作为参考输入，但本 skill 的 P0 深度阅读阶段仍然是强制的，不可跳过。

### 流程降级机制

标准流程是完整 7 阶段，但允许根据**任务复杂度**降级，降级不等于放弃质量：

| 降级级别 | 适用场景 | 可精简的阶段 | 不可降级的底线 |
|----------|---------|-------------|---------------|
| 完整流程 | 新增模块、重构、涉及共享接口变更 | 无 | 全部 P0-P6 |
| 标准流程 | 新增页面、修改涉及 ≤3 文件 | P0/P1 可合并确认 | P0 必读 + P3 + P4 + P5 + P6 |
| 简化流程 | 改动 ≤3 行、无新文件、无依赖变更 | P1/P2 可省略，P6 精简 | P0 + P3 + P4 |
| 紧急流程 | 线上 hotfix | P1 可跳过，P2 精简 | P3 编码标准不降 + P4 至少做类型检查 |

**降级原则**：
- 编码标准（P3.3 禁止项/强制项）在任何降级级别下都不可降低
- 编译验证（P4）在任何降级级别下都不可跳过
- 降级后必须在交付输出中标注"本次执行了 X 级降级流程"及降级原因
- 紧急流程完成后，后续必须补做完整的 P0-P5 流程

```
 ┌───────────────────────────────────────────────────────────────────┐

 │       ↓                  ↓               ↓                       │
 │  [质量门禁-0]       [质量门禁-1]     [质量门禁-2]                 │
 │                                            ↓                      │

 │                                        ↓                         │

 └───────────────────────────────────────────────────────────────────┘
```

## P0: 深度阅读（强制第一阶段，禁止跳过）

**目的**：在写任何代码之前，充分理解项目的所有相关上下文。不读不写，这是铁律。

### P0.1 项目结构扫描

扫描项目根目录和 `src/` 目录，获取全貌：
- 构建工具和配置（`vite.config.*` / `webpack.config.*` / `nuxt.config.*`）
- 依赖版本（`package.json` 中的 vue、vue-router、pinia、UI 框架版本）
- TypeScript 配置（`tsconfig.json` — strict 模式? 路径别名?）
- Lint/格式化配置（`.eslintrc.*` / `.prettierrc.*` / `eslint.config.*`）
- 环境变量文件（`.env*` — 了解 API 地址、功能开关）

### P0.2 目标区域深度阅读

根据任务类型，**必须完整阅读**以下文件（不是浏览，是逐行理解）：

**新增功能时读取：**
- 目标模块的目录下所有文件
- 目标模块的 `index.ts`（公共接口）
- 目标模块的 `types.ts`（类型定义）
- 目标模块的 `composables/*.ts`（已有逻辑）
- 路由配置中与目标模块相关的部分
- 目标页面要使用的 API 接口定义

**修改已有功能时读取：**
- 要修改的文件的**完整内容**（不是只读改动的部分）
- 该文件 import 的所有直接依赖（至少读取接口签名部分）
- 该文件被哪些其他文件引用（grep 查找所有 import 了该文件的地方）
- 该文件关联的 composable/store/API 文件

**重构时读取：**
- 涉及重构的所有文件的完整内容
- 每个文件的调用方和被调用方（完整的依赖图）
- 相关的类型定义文件
- 相关的测试文件（如果存在）

### P0.3 已有复用资源盘点

在阅读过程中，同步记录：
- 已有的通用组件（`shared/components/` 或 `components/`）— 列出可复用的
- 已有的 composables（`composables/` 或 `hooks/`）— 列出可复用的
- 已有的工具函数（`utils/`）— 列出可复用的
- 已有的类型定义（`types/`）— 列出可扩展/复用的
- 已有的 API 函数（`api/`）— 列出可复用的
- **已有同类页面**（`views/`）— 找到与新需求同类型的已有页面，记录其布局结构、使用的组件、交互模式，作为新页面的参考模板

### 质量门禁 P0（必须全部通过才能进入 P1）

- [ ] 已读取 `package.json` 确认技术栈版本
- [ ] 已读取构建配置确认项目约束
- [ ] 已完整阅读所有直接相关的源文件
- [ ] 已确认被修改文件的所有调用方
- [ ] 已盘点可复用的现有组件/composable/工具函数/类型
- [ ] 已找到同类页面作为参考模板（新增页面时）
- [ ] 能用一句话准确描述当前需求的本质

> 如果 P0 未通过：**禁止进入 P1**，继续阅读直到满足条件。

### P0 用户确认点：需求澄清

P0 完成后，**必须向用户展示以下内容并等待确认**：

```
【需求理解】
我理解您需要的是：（用用户能听懂的自然语言描述需求本质）

【任务复杂度】
判定为：（简单/标准/复杂/紧急）
理由：（改动行数、涉及文件数、是否涉及共享接口等）

【已发现的不明确点】
在阅读项目代码后，以下几点我需要向您确认：
1. （具体的不明确点 + 你的理解 + 请用户选择或补充）
2. ...

【已识别的现有资源】
- 可复用组件：（列出）
- 可复用 composable：（列出）
- 相关已有代码：（列出）
- 参考页面：（列出同类已有页面）

请确认我的理解是否正确，或补充您的要求。
```

- 如果需求有歧义，**必须提问**，禁止猜测用户的意图
- 用户确认"可以"/"没问题"后才能进入 P1
- 用户补充了新信息 → 回到 P0 补充阅读，重新确认

## P1: 全局分析

**目的**：基于 P0 的阅读结果，建立全局认知，确定变更的影响范围和约束。

### P1.1 技术栈锁定

根据 P0 扫描结果，锁定当前项目的技术决策（不是推荐，是项目已有的事实）：

| 维度 | 检测方式 | 锁定决策 |
|------|----------|----------|
| 组件风格 | grep `<script setup` vs `export default` | Composition API `<script setup>` 或 Options API |
| 语言 | `.vue` 文件中 `lang="ts"` 出现率 | TypeScript 或 JavaScript |
| 状态管理 | package.json dependencies | Pinia / Vuex / 无 |
| CSS 方案 | `<style>` 标签内容特征 | Tailwind / Scoped CSS / CSS Modules / 其他 |
| UI 框架 | package.json dependencies | Element Plus / Ant Design Vue / Naive UI / 自定义 / 无 |
| 构建工具 | 配置文件类型 | Vite / Webpack / Nuxt |

### P1.2 影响范围分析

确定最小变更集：
```
必须新建的文件（列表）:
  └── 列出文件路径 + 一句话说明用途

必须修改的文件（列表）:
  └── 列出文件路径 + 说明修改内容 + 确认不影响其他调用方

可能影响的文件（需验证）:
  └── 列出文件路径 + 需要验证的点

明确不需要修改的文件:
  └── 列出可能被误改但实际不需要动的文件
```

### P1.3 风险评估

- 是否涉及共享组件的接口变更？（影响所有调用方）
- 是否涉及 store 的结构变更？（影响所有消费者）
- 是否涉及路由配置变更？（影响导航和权限）
- 是否涉及类型定义变更？（影响所有引用方）
- 是否有潜在的破坏性变更？（列出并给出保护措施）

### 质量门禁 P1

- [ ] 技术栈已锁定，所有决策基于项目事实
- [ ] 变更范围清单已完成，新建/修改/验证文件已明确列出
- [ ] 风险点已识别并有对应保护措施
- [ ] 确认变更不会破坏任何已有功能

### P1 用户确认点：变更范围确认

P1 完成后，**必须向用户展示以下内容并等待确认**：

```
【技术栈确认】
项目使用：（列出锁定的技术栈）

【变更范围】
需要新建的文件：（列出路径 + 用途）
需要修改的文件：（列出路径 + 修改内容）
不需要改动的文件：（列出排除项）

【风险评估】
潜在风险点：（列出）→ 保护措施：（说明）

【需要确认的问题】
（如果变更范围或影响有任何不确定的地方，在这里提问）

请确认变更范围是否正确，或补充/调整。
```

- 用户确认后才能进入 P2
- 如果用户有调整 → 回到 P0/P1 补充分析

## P2: 方案设计

**目的**：在编码前完成所有设计决策，编码阶段只负责翻译设计为代码。

### P2.1 文件级设计

对每个要新建或修改的文件，确定：
- 文件职责（一句话）
- 导出接口（props/emits/函数签名/类型）
- 内部依赖（import 什么）
- 被谁依赖（谁 import 它）

### P2.2 组件/函数级设计

对每个组件和核心函数，确定：
- 输入：接收什么数据（props/参数）
- 输出：产生什么（emits/返回值/DOM）
- 副作用：是否有 API 调用、DOM 操作、全局状态变更
- 状态：内部管理哪些响应式数据
- 复用：是否提取 composable 或使用已有的

### P2.3 数据流设计

画出数据从获取到展示的完整路径：
```
API 层 → Composable 层 → 组件 props → 模板渲染
                ↕
           Pinia Store（如需全局共享）
```

### P2.4 设计决策记录

对每个非平凡的设计选择，记录：
- 选择了什么方案
- 为什么选择这个方案（而不是其他方案）
- 这个决策的约束和代价

### 质量门禁 P2

- [ ] 每个新建/修改文件有明确的职责定义
- [ ] 组件/函数接口已设计（输入、输出、副作用）
- [ ] 数据流已明确
- [ ] 所有设计决策有理由
- [ ] 方案与 P1 的变更范围一致（无遗漏无膨胀）

### P2 用户确认点：方案设计确认（进入编码前的最后一道确认）

P2 完成后，**必须向用户展示以下内容并等待确认**：

```
【方案概述】
（简明扼要描述整体实现方案）

【文件清单与职责】
文件路径 — 职责描述
...

【关键设计决策】
- 决策 1：选择了 X 方案，原因是 Y
- 决策 2：...

【数据流】
（简要描述数据从获取到展示的流向）

【待确认问题】
（设计中需要用户拍板的选择，如：多种方案对比、是否需要某个功能、是否接受某个限制）

确认无误后我将开始编码。如有调整请告知。
```

- 用户确认后才能进入 P3 编码
- 这是编码前的最后一道防线，确认后才开始动手
- 如果用户有调整 → 更新 P2 设计并重新确认

## P3: 编码实现

**目的**：将 P2 的设计翻译为代码。此阶段只做翻译，不做设计变更。

### P3.1 编码顺序（严格按依赖方向）

```
1. 类型定义 → 先确定数据结构，所有后续代码依赖于此
2. API 函数 → 确定接口调用，composable 依赖于此
3. Composable → 确定业务逻辑，组件依赖于此
4. 子组件 → 确定展示单元，父组件依赖于此
5. 父组件/页面 → 组装所有部分
6. 路由/Store 注册 → 将新模块接入系统（如需要）
```

每完成一个文件，立即做 **P3.2 微自检** 后再写下一个。

### P3.2 文件级微自检（每个文件写完后立即执行）

对刚写完的每个文件：

1. **编译预检**：在脑中模拟 TypeScript 编译，检查类型是否正确、导入路径是否存在
2. **接口一致性**：检查 props/emits/函数签名是否与 P2 设计一致
3. **依赖正确性**：所有 import 是否指向正确的路径和导出
4. **规范符合性**：命名、注释、代码结构是否符合 `references/code-standards.md`
5. **最小改动验证**：如果是修改已有文件，确认只改了必要的行

### P3.3 编码规则

**禁止项（违反任何一条都不可交付）：**
- 禁止使用 `any` 类型（用 `unknown` 或具体类型替代）
- 禁止 `@ts-ignore`（用 `@ts-expect-error` + 注释说明原因）
- 禁止非空断言 `!`（除非能证明安全且注释原因）
- 禁止 `v-for` + `v-if` 在同一元素上
- 禁止直接修改 props
- 禁止在 `watch` 中做可以 `computed` 的事
- 禁止硬编码魔法值
- 禁止在模板中写复杂表达式（超过两个运算符）
- 禁止 `console.log` / `debugger`（调试完成后必须移除）
- 禁止注释掉的代码（用 git 管理历史）
- **禁止手写项目中已有通用组件能实现的 UI**（如已有 `DataTable` 就不用原生 table + 手写样式）
- **禁止硬编码颜色、间距、字号等视觉值**（必须使用 CSS 变量或 UI 框架 token）
- **禁止同一类型页面使用不同的交互模式**（如列表页的新增操作，统一用弹窗就全用弹窗，统一用抽屉就全用抽屉）

**强制项：**
- 所有 props 必须有 TypeScript interface 定义
- 所有 emits 必须有类型签名
- 组件默认 scoped 样式
- 引用类型 props 默认值用工厂函数 `() => []`
- API 响应有完整类型定义
- 导入按规范分组排序（type → Vue 核心 → 第三方 → 内部 composables → 内部 utils → 组件）
- 文件内声明顺序符合规范（types → imports → props → emits → refs → computed → watch → 生命周期 → 方法）
- **新页面必须参考 P0 中记录的同类页面模板**，保持布局结构和代码组织一致

### 质量门禁 P3

- [ ] 所有文件按依赖顺序编写
- [ ] 每个文件通过了微自检
- [ ] 无禁止项
- [ ] 所有强制项已满足
- [ ] 代码与 P2 设计方案一致

## P4: 编译验证与自动修复（强制，禁止跳过）

**目的**：通过工具验证代码正确性，发现并自动修复所有问题。此阶段是一个**修复循环**，
直到所有检查全部通过才退出。

### P4.1 执行验证命令

根据项目配置，依次执行（自动检测可用的命令）：

```bash
# 1. TypeScript 类型检查（如果有 tsconfig.json）
npx vue-tsc --noEmit
# 或
npx tsc --noEmit

# 2. ESLint 检查（如果有 eslint 配置）
npx eslint src/ --ext .vue,.ts,.tsx

# 3. 单元测试（如果有 vitest 配置）
npx vitest run

# 4. 构建检查（验证最终产物）
npm run build
# 或
npx vite build
```

### P4.2 错误分类与处理策略

**致命错误（必须修复，否则阻塞）：**
- TypeScript 类型错误 → 分析类型不匹配原因，修正类型定义或实现
- 编译错误（导入路径、语法错误）→ 检查文件是否存在、路径是否正确、语法是否合法
- 构建失败 → 根据构建日志定位问题并修复

**重要警告（应该修复）：**
- ESLint 错误 → 根据规则调整代码
- 未使用的变量/导入 → 清理
- 缺失返回类型 → 补充

**建议优化（可以修复）：**
- ESLint 警告 → 评估是否值得修复
- 性能提示 → 记录但不阻塞

### P4.3 自动修复循环

```
修复循环（最多 5 轮）：
  ┌──────────────────────────────────┐
  │  1. 执行验证命令                   │
  │  2. 收集所有错误和警告             │
  │  3. 如果全部通过 → 退出循环        │
  │  4. 按优先级排序错误               │
  │     - 致命错误优先                 │
  │     - 同一文件错误合并处理         │
  │  5. 分析根因，修复代码             │
  │  6. 验证修复未引入新问题           │
  │  7. 回到步骤 1                    │
  │                                  │
  │  如果 5 轮后仍有致命错误：          │
  │  → 报告给用户，说明剩余问题        │
  └──────────────────────────────────┘
```

### P4.4 修复原则

- **根因修复**：修复根本原因而非症状。类型错误不要用 `as any` 掩盖
- **最小修复**：只改需要改的，修复一个错误不要引入新的重构
- **验证修复**：每次修复后确认原问题已解决且未引入新问题
- **不降低标准**：禁止通过 `@ts-ignore`、`any`、`// eslint-disable` 等方式绕过检查

### 质量门禁 P4

- [ ] TypeScript 类型检查零错误
- [ ] ESLint 零错误（警告可接受但需记录）
- [ ] 构建成功，无编译错误
- [ ] 修复循环未超过 5 轮
- [ ] 所有修复都是根因修复，无绕过行为

> 如果 P4 未通过且超过 5 轮：**暂停并向用户报告**，说明具体问题和建议。

## P5: 需求回验（强制，编译通过后必须执行）

**目的**：代码编译通过不代表需求已完成。此阶段要求回到原始需求本身，逐条验证每个需求点
是否被正确实现。不满足则自动回退修复，形成闭环。

这是整个流水线中最关键的"守门员"——前四个阶段保证代码"写得对"，P5 保证代码"做对了事"。

### P5.1 需求拆解与清单化

重新阅读用户的原始需求，将其拆解为可验证的原子化需求点：


对每个需求点，执行以下验证：

```

```

## 核心设计原则

以下原则在所有阶段都适用，是每一个编码决策的判断标准。

### UI 一致性原则（铁律级）

所有页面的视觉风格、交互模式、布局结构必须保持统一，禁止凭个人感觉随意编写页面。

- **组件复用优先**：页面中需要的 UI 元素（表格、表单、弹窗、卡片、搜索栏等），必须优先使用项目已有的通用组件或 UI 框架组件，禁止从零手写一个风格不同的新组件
- **布局结构对齐**：同类型页面（如列表页、详情页、表单页）必须沿用相同的布局骨架（header + content + footer 的结构和间距），禁止一个页面紧凑一个页面松散
- **交互模式统一**：相同的操作（新增、编辑、删除、搜索、分页）在不同页面中的触发方式和反馈方式必须一致，禁止一个页面用弹窗另一个页面用抽屉做同样的事
- **样式变量强制使用**：颜色、间距、字号、圆角、阴影等必须使用项目定义的 CSS 变量或 UI 框架 token，禁止硬编码任何视觉值
- **页面模板参考**：新建页面时，必须先找到项目中已有的同类页面作为参考模板，保持一致的页面结构和代码组织方式

**判断标准**：新页面与已有页面放在一起，如果用户能一眼看出"风格不一样"，就是违规。

### 最小改动原则（第一优先级）

每次变更必须是最小必要改动：
- 改动直接服务于当前需求？不是就去掉
- 顺手"优化"了不相关代码？还原
- 引入了当前不需要的抽象？简化
- 能修改已有代码而非新增来解决问题？优先修改

### 高内聚低耦合

- **内聚**：一个模块/组件/函数只负责一件事
- **耦合**：通过明确接口通信，不直接访问彼此内部
- **依赖方向**：单向，子不依赖父，业务不依赖 UI

### 自动化驱动

- 需求分析和方案设计阶段：**必须与用户确认**，不擅自行动
- 技术选择和风格适配：自动完成，无需人工确认
- 编码后：自动验证、自动修复、自动优化
- 遇到无法自动解决的问题才上报用户

| P4 编译验证 | 测试/lint 类 skill | 自动化验证和修复 |
| P6 最终审查 | 代码审查类 skill | 获取代码质量建议 |

### 使用原则

- **优先使用已有 skill**：如果环境中存在能完成某个子任务的 skill，优先使用它
- **不重复造轮子**：skill 已经封装好的能力不需要手动实现
- **合理组合**：多个 skill 可以组合使用，但要注意上下文衔接
- **不强行使用**：如果没有合适的 skill 或 skill 不适用，按本规范手动处理

进入对应阶段时，**自动加载**相关参考文件（不需要用户触发，也不需要等到遇到问题才翻阅）。
分为**核心文件**（每次任务都会用到）和**按需文件**（只在特定任务类型时需要）：

### 核心文件（每次任务加载）
