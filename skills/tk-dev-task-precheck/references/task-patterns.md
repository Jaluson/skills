# 任务类型识别模式

本文档帮助 AI 在 T0 阶段快速、准确地识别用户编码任务的类型。

## 一、任务分类体系

### 1. 前端开发任务

**强信号关键词**：
- 页面、组件、表单、弹窗、对话框、抽屉、表格、列表
- 路由、导航、菜单、标签页
- 样式、布局、响应式、主题
- 交互、动画、拖拽、拖放
- 图表、可视化、ECharts
- 上传、下载、预览
- 登录页、注册页、详情页、列表页
- 按钮、输入框、选择器、日期选择

**隐含前端信号**（需结合上下文判断）：
- "展示一下" "显示" "看效果"
- 涉及 .vue / .jsx / .tsx / .html / .css / .scss 文件
- 涉及浏览器、响应式、适配

### 2. 后端开发任务

**强信号关键词**：
- 接口、API、REST、CRUD、增删改查
- 数据库、表、字段、索引、SQL
- 服务、Service、Controller、Mapper、Repository
- 实体、DTO、VO、Model
- 事务、锁、并发、缓存
- 鉴权、权限、角色、登录认证
- 中间件、Redis、MQ、消息队列
- 定时任务、批处理、导入导出
- 校验、参数校验、数据校验

**隐含后端信号**：
- "处理逻辑" "业务规则" "数据存储"
- 涉及 .java / .py / .go / .xml 文件
- 涉及数据库、服务器

### 3. 全栈开发任务

**强信号关键词**：
- 完整功能、端到端、前后端联调
- "从数据库到页面" "从接口到展示"
- 涉及前后端同时存在的需求

**典型模式**：
- "做一个用户管理功能" → 通常意味着后端 CRUD 接口 + 前端管理页面
- "新增一个 XX 模块" → 通常意味着后端服务 + 前端页面
- "实现 XX 功能" → 需要分析是否涉及前后端

### 4. Bug 修复任务

**强信号关键词**：
- 报错、异常、错误、不工作、不对、有问题
- 404、500、空指针、类型错误
- 修复、修 bug、解决问题
- 崩溃、卡死、闪退、白屏
- 与预期不符、行为异常

**分类**：
- 前端 Bug → UI 展示问题、交互异常、样式错乱
- 后端 Bug → 接口报错、数据不一致、逻辑错误
- 环境 Bug → 构建失败、运行异常、依赖冲突

### 5. 重构任务

**强信号关键词**：
- 重构、优化、拆分、合并、迁移
- 提取公共、抽象、封装
- 代码整理、代码清洁
- 性能优化、加载优化、渲染优化
- 升级依赖、技术栈迁移

### 6. 配置变更任务

**强信号关键词**：
- 配置、环境变量、部署、CI/CD
- Docker、容器化、K8s
- Nginx、代理、域名
- 热更新、打包、构建
- 跨域、代理配置

### 7. 测试任务

**强信号关键词**：
- 写个单元测试、测试用例、测试覆盖率
- 修复失败的测试、测试挂了、测试没过
- mock 一下、stub、spy
- 集成测试、端到端测试、E2E
- TDD、测试驱动开发
- 测试套件、测试夹具、fixture

**隐含测试信号**：
- "验证一下" "确保没问题"
- 涉及 .test. / .spec. / _test. / Test 后缀文件
- 涉及覆盖率报告、CI 测试流水线

### 8. 数据库迁移任务

**强信号关键词**：
- 加个字段、建个表、写个 migration
- 改一下表结构、数据库变更、数据迁移
- 表结构、列类型、索引调整
- 数据清洗、数据转换、数据导入
- Flyway、Liquibase、Alembic、Prisma migrate

**隐含数据库迁移信号**：
- "更新一下模型" 且项目使用 ORM
- "新增一个实体" 可能需要同步建表
- 涉及 .sql 文件或 migration 目录

### 9. 代码生成任务

**强信号关键词**：
- 生成、创建、初始化、脚手架
- 新建项目、新建模块
- 代码模板、代码生成器
- "帮我搭一个" "初始化一个"

## 二、歧义场景处理

### 高歧义场景

| 用户表述 | 可能理解 A | 可能理解 B | 处理策略 |
|----------|-----------|-----------|---------|
| "加个字段" | 数据库加字段 + 后端实体修改 | 前端表单加个输入项 | 提问确认 |
| "改一下那个功能" | 改前端展示逻辑 | 改后端业务逻辑 | 提问确认 |
| "做个导出" | 前端导出按钮 + 后端导出接口 | 只做后端导出接口 | 提问确认 |
| "优化一下" | 性能优化（可能涉及前后端） | 代码重构 | 提问确认 |
| "加个按钮" | 纯前端 UI 操作 | 前端按钮 + 后端对应接口 | 提问确认 |
| "写个管理页面" | 只写前端页面（后端已有） | 前后端都写 | 提问确认 |
| "写个接口" | Java/TypeScript 的 interface 定义 | REST API 端点实现 | 提问确认 |
| "加个权限" | 后端 RBAC 配置 | 前端路由守卫，或两者都要 | 提问确认 |
| "改一下样式" | 纯前端 CSS/样式修改 | 涉及后端模板渲染（如 Thymeleaf/JSP） | 提问确认 |
| "部署一下" | 配置变更 | CI/CD 流水线，或环境搭建 | 提问确认 |
| "加个缓存" | 前端缓存（localStorage/Pinia） | 后端缓存（Redis），或 HTTP 缓存配置 | 提问确认 |

### 处理原则

1. **宁可多问一句，不可猜错方向** — 修一分钟的返工不如多十秒的确认
2. **提供选项而非开放式问题** — "您是只做前端还是前后端都要？" 比 "请具体说说" 更高效
3. **结合项目上下文辅助判断** — 如果用户说"加个按钮"，且项目是纯前端项目，那大概率是纯前端操作

## 三、紧急度判断

| 紧急度 | 信号 | 行为 |
|--------|------|------|
| 低 | "有空的话" "以后再说" "看看能不能" | 正常走完前置流程 |
| 中 | 默认，无特殊信号 | 正常走完前置流程 |
| 高 | "紧急" "尽快" "线上问题" "马上" | 前置流程可适当简化，但 T0 和 T3 不可跳过 |

注意：即使紧急，T0（任务分析）和 T3（路由分发）也不可跳过。可以简化的是 T1（项目扫描）和 T2（环境检查）的深度。

### 紧急度对交接的影响

紧急度应写入交接包，供下游 skill 调整工作策略：

| 紧急度 | 交接包标记 | 下游 skill 行为调整 |
|--------|-----------|-------------------|
| 高 | `urgency: high` | 下游 skill 可跳过非必要的分析步骤（如代码风格审查、详细影响范围评估），优先给出可执行方案 |
| 中 | `urgency: normal`（或省略） | 下游 skill 按正常流程执行 |
| 低 | `urgency: low` | 下游 skill 可进行更深入的分析和优化建议 |

高紧急时，交接包应包含明确的紧急度标记，下游 skill 据此简化非必要分析步骤，聚焦于快速解决问题。
