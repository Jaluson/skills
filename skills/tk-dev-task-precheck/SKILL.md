# 开发任务前置守门员 (tk-dev-task-precheck)

## 前置条件：环境检测（强制）

在执行任何编码任务之前，**必须先完成环境检测**：

```
┌─────────────────────────────────────────────────────┐
│  T0: 任务意图分析（用户交互）                          │
│       ↓                                              │
│  T1: 项目上下文预扫描（并行可加速）                     │
│       ↓                                              │
│  T2: 环境就绪检查（强制调用 tk-dev-env-manager）        │
│       ├─ 调用 tk-dev-env-manager 执行完整检测          │
│       ├─ 环境就绪 → 继续 T3                          │
│       └─ 环境缺失 → 配置完成后再继续                   │
│       ↓                                              │
│  T3: Skill 路由分发（用户交互）                        │
│       ↓                                              │
│  T4: 上下文交接 → 下游 skill 执行编码                  │
└─────────────────────────────────────────────────────┘
```

**未通过环境检测的任务不得进入 T3 路由分发阶段。**

## 这个 skill 解决什么问题

现有 skill 各自独立运行，每次触发时都要从零开始扫描项目、识别技术栈、检查环境。
本 skill 是**所有编码任务的统一入口**，在下游 skill 启动前完成：
1. 分析用户到底要做什么
2. 预扫描项目，提前建好上下文
3. 确认环境就绪
4. 把任务交给正确的 skill，并把预扫描结果一并交接

## 四条铁律

**铁律一：先分析后行动，绝不跳过前置流程。**
即使用户催促说"直接开始""别废话"，也必须完成 T0-T3 的前置分析。
前置分析的目的不是拖延，而是避免后续返工——方向错了，跑得越快浪费越多。

**铁律二：只准备不执行，该交接就交接。**
本 skill 的职责是"准备"而不是"执行"。完成前置分析后，必须将任务交接给
对应的下游 skill（tk-vue3-dev-standards、tk-springboot-dev-standards、tk-database-design-standards 等），
让它们按照自己的流水线执行。**禁止**在本 skill 中启动子 agent 来实现编码——
子 agent 只能用于扫描和检测，不能用于编写代码。

**铁律三：不明确就问，不猜测用户意图。**
如果无法确定任务的性质（前端还是后端？新建还是修改？涉及哪个模块？），
先检查项目上下文（T1）看能否消除歧义。如果项目上下文也无法确定，
必须向用户提问，而不是凭猜测选择下游 skill。

**提问规则：**
- **有选择时提供选项**：如果能想到几种合理的方案或理解方式，在提问时直接提供选项（如"A / B / C"），
  让用户可以快速选择，而不是让用户自己组织语言输入
- **多问题逐个回答**：如果一次分析中发现多个需要确认的点，**逐个提出**，
  等用户回答一个后再提下一个，避免一次性罗列所有问题让用户一次性回答
- **用一句话确认理解**：先用自己的话复述理解，再用提问确认具体细节

**铁律四：阶段内可并行，阶段间必须串行。**
T0 → T1 → T2 → T3 → T4 之间有严格的数据依赖，不可跨阶段并行。
但在每个阶段**内部**，独立的扫描、检测任务可以创建子 agent 并行执行。
子 agent 必须有明确的输入、输出和职责边界，且只负责只读操作（扫描/检测），
不负责写操作（编码/修改文件）。

**铁律五：变更后必须验证。** 任何文件变更（新建/修改/删除配置文件、交接包文档等）完成后，
必须重新执行环境检测验证变更是否正确，并重新确认是否满足用户的原始需求。
环境检测失败或需求不满足的变更不得交付。

**铁律六：提问后必须重新确认。** 向用户提问获取回答后，必须基于回答重新出具完整方案，并将该方案以菜单选项形式交给用户最终确认。用户选择确认选项后，方可进入下一步。未获得用户确认的方案不得执行。

**铁律七：方案必须包含详细的改动信息。** 提交给用户确认的方案中，对于每个文件变更必须包含：
- **前端文件**：完整路径+文件名、文件类型、所属层级、核心职责、改动逻辑
- **后端文件**：完整包路径+类名或文件路径、层级、核心职责、改动逻辑
- **数据库表**：表名称、表用途、字段清单、索引设计
- **配置文件**：文件路径、配置项变更内容

示例格式：
```
【文件变更清单】
前端：
  - src/views/user/UserList.vue                        — 用户列表页，包含表格、搜索、分页
  - src/api/user.ts                                    — 用户模块 API 接口定义
  - src/router/index.ts                                — 新增 /user 路由配置

后端：
  - com.example.user.controller.UserController          — 用户管理 Controller，处理 CRUD 接口
  - com.example.user.service.UserService                — 用户业务逻辑
  - com.example.user.entity.User                        — 用户实体类

数据库：
  - sys_user 表                                        — 新增字段 address_id BIGINT

配置：
  - application.yml                                    — 新增 upload.max-size 配置项
```

禁止仅用"用户管理功能"等模糊描述，必须逐项列出具体文件路径和改动逻辑。

## 子 Agent 协作规范

### 什么可以并行

```
可以并行的条件（全部满足才并行）：
  ✅ 在同一阶段内部（如都在 T1 内，或都在 T2 内）
  ✅ 任务之间无数据依赖（A 的输入不依赖 B 的输出）
  ✅ 子 agent 只做只读操作（扫描文件、检测工具，不修改任何文件）
  ✅ 任务不需要用户交互（子 agent 无法与用户对话）
  ✅ 任务有明确的输出格式（能定义清楚"返回什么"）
```

### 什么不能并行

- 跨阶段的操作（T1 的扫描结果未出就不能启动 T2）
- 需要向用户提问或等待确认的环节
- 涉及修改文件的操作

### 子 Agent 使用约束

| 约束 | 说明 |
|------|------|
| 职责明确 | 创建时用一句话说明"负责做什么" |
| 输出明确 | 告诉它"完成后返回什么格式" |
| 只读不写 | 前置阶段的子 agent 不修改任何文件 |
| 数量可控 | 一般不超过 5 个并行 |
| 主 agent 汇总 | 子 agent 返回结果由主 agent 统一判断 |

## 五阶段前置流程

```
用户发起编码任务
       |
       v
  ┌───────────────────────────────────────────────────────┐
  │ T0: 任务意图分析                                       │
  │     ┌─ 模糊？→ 提问用户 → 补充信息 → 重新分析 ─┐      │
  │     └─ 明确 ────────────────────────────────────┘      │
  │                        ↓                               │
  │ T1: 项目上下文预扫描（⚡ 内部可并行）                    │
  │                        ↓                               │
  │ T2: 环境就绪检查（委托 tk-dev-env-manager）                │
  │     ┌─ 缺失且影响任务 → 引导配置 → 重新检查 ─┐        │
  │     └─ 就绪或已记录 ─────────────────────────┘        │
  │                        ↓                               │
  │ T3: Skill 路由分发                                     │
  │     ┌─ 用户有调整 → 回退修正 ──────────────┐          │
  │     └─ 用户确认 ───────────────────────────┘          │
  │                        ↓                               │
  │ T4: 上下文交接 → 打包交接给下游 skill → 本 skill 结束   │
  └───────────────────────────────────────────────────────┘
```

## T0: 任务意图分析

**目的**：准确理解用户要做什么，判断任务的性质和范围。

> 📖 详细的任务识别关键词和歧义场景，参见 `references/task-patterns.md`。

### T0.1 需求分类

将用户的编码需求归入以下类别之一（或多个）：

| 任务大类 | 典型关键词 |
|----------|-----------|
| 前端开发 | 页面、组件、表单、弹窗、路由、样式 |
| 后端开发 | 接口、API、服务、实体、数据库、SQL |
| 全栈开发 | 完整功能、前后端联调 |
| Bug 修复 | 报错、异常、不工作、修复 |
| 重构 | 重构、优化、拆分、迁移 |
| 配置变更 | 配置、部署、CI/CD、Docker |
| 测试任务 | 单元测试、测试覆盖率、mock |
| 数据库设计 | 建表、加字段、设计索引、表结构、DDL、migration |
| 代码生成 | 初始化、脚手架、新建模块 |

### T0.2 技术领域判断

根据需求内容判断技术领域：

- **前端**：UI 展示、用户交互 → 可能触发 `tk-vue3-dev-standards`
- **后端**：数据存储、业务逻辑 → 可能触发 `tk-springboot-dev-standards`
- **数据库**：表结构设计、DDL、索引、migration → 可能触发 `tk-database-design-standards`
- **环境**：构建、运行、工具链 → 可能触发 `tk-dev-env-manager`
- **多领域**：同时涉及前后端 → 可能需要多个 skill 协作

### T0.3 模糊需求处理

如果用户的需求表述模糊，**必须提问**。

**提问原则：**
- **有选择时提供选项**：能想到 2 种及以上理解方式时，直接在提问中列出选项（A / B / C），
  让用户可以点选或回复选项字母，而不是让用户自己组织语言描述
- **逐个提问**：如果一次发现多个待确认点，**先提第一个**，等用户回答后再提下一个；
  避免一次性抛出"请确认以下三点"让用户一次性回答所有问题
- **先复述再提问**：先用一句话复述你理解到的内容，再用"具体是哪种情况？"的方式提问

**提问格式示例：**

```
【需求确认】
我理解您要做的是：给用户列表页面添加一个导出功能。

请问您希望的导出方式是：
  A. 导出为 Excel 文件
  B. 导出为 CSV 文件
  C. 直接打印

请回复选项字母。
```

**判断标准**：对一个需求点能想到两种及以上理解方式时，必须提问，并提供选项。

### 质量门禁 T0

- [ ] 用户需求已被归入正确的任务类别
- [ ] 技术领域已判断（前端/后端/全栈/环境）
- [ ] 有歧义的需求已向用户确认并获得明确答复
- [ ] 能用一句话描述任务的核心目标

**未通过时的处理**：回到 T0.3 继续向用户提问，直到所有歧义消除。禁止带着模糊理解进入 T1。

## T1: 项目上下文预扫描

**目的**：快速扫描项目，为下游 skill 准备好"项目概况包"。

> 📖 详细的扫描清单、字段提取规则和多模块策略，参见 `references/project-scan-checklist.md`。

### T1 并行化策略

当项目有 3 个以上配置文件需要读取时，推荐并行：

```
并行组 1 — 配置文件读取：
  Agent A: 读取 pom.xml → 提取后端技术栈
  Agent B: 读取 package.json → 提取前端技术栈
  Agent C: 读取辅助配置（tsconfig / vite.config / application.yml 等）

并行组 2 — 资源盘点（组 1 完成后）：
  Agent D: 扫描前端资源目录
  Agent E: 扫描后端资源目录
```

简单项目（配置文件 ≤ 2 个）直接串行，无需子 agent。

### T1.1 项目类型识别

扫描项目根目录的标志性文件：`pom.xml`、`build.gradle`、`package.json`、
`requirements.txt`、`pyproject.toml`、`go.mod`、`Cargo.toml`、`.csproj`、
`composer.json`、`Gemfile` 等。一个项目可能同时存在多个标志性文件。

### T1.2 技术栈快照

从配置文件中提取关键信息（快速扫描，不逐行阅读）：

**Java 项目**：Spring Boot 版本、Java 版本、ORM、数据库驱动、中间件
**前端项目**：框架及版本、UI 组件库、构建工具、TypeScript、路由、状态管理
**Gradle 项目**：从 `sourceCompatibility`/`jvmTarget` 提取 Java 版本，从 `dependencies` 提取关键依赖

### T1.3 目录结构概览

扫描顶层目录，生成简洁的项目地图。只做顶层扫描，不深入子目录。

### T1.4 现有资源盘点

快速识别可复用资源（只扫文件名列表，不读内容）：
前端：components/、views/、composables/、api/、utils/、types/、stores/、layouts/
后端：controller/、service/、mapper/、entity/、dto/、vo/、config/、common/

### 质量门禁 T1

- [ ] 项目类型已识别
- [ ] 技术栈快照已生成（含版本号）
- [ ] 目录结构概览已完成
- [ ] 现有资源已快速盘点

**未通过时的处理**：补充扫描缺失的部分。如果配置文件读取失败或内容异常，记录并在交接包中标注。

## T2: 环境就绪检查

**目的**：确认开发环境可用，避免编码到一半才发现工具链缺失。

### 核心策略：委托 tk-dev-env-manager

环境检测**优先委托给 `tk-dev-env-manager` skill** 执行，而不是自己做浅层检测。
原因：
- `tk-dev-env-manager` 能做版本校验（不仅是工具可达性检查）
- `tk-dev-env-manager` 能读取 `.claude/project-env/.env` 配置，使用项目指定的工具路径
- `tk-dev-env-manager` 有完整的环境配置引导能力

### T2.1 调用 tk-dev-env-manager

```
向 tk-dev-env-manager 传递的信息（避免重复检测）：
  - T1 已识别的项目类型和技术栈
  - T1 已提取的版本要求（如 pom.xml 中的 Java 版本）
  - 请求从 E2（配置文件读取）开始，跳过 E0（项目类型检测）和 E1（环境需求解析）
  - skip_e0_e1: true（显式标记，确保 tk-dev-env-manager 不会重复执行 E0/E1）

从 tk-dev-env-manager 获取的返回：
  - 各工具的安装状态和版本号
  - 是否满足项目要求的版本
  - 缺失工具清单
```

### T2.2 状态判定

```
环境状态：✅ 就绪 / ⚠️ 部分就绪 / ❌ 缺失

如果全部就绪：
  → 记录环境状态，继续 T3

如果部分就绪或缺失：
  → 判断缺失项是否影响当前任务
     - 不影响（如后端任务缺 Node）→ 记录但不阻塞，继续 T3
     - 影响（如后端任务缺 JDK）→ 由 tk-dev-env-manager 引导配置
  → 用户确认环境就绪后，继续 T3
```

### T2.3 tk-dev-env-manager 不可用时的降级方案

如果 `tk-dev-env-manager` 不可用，使用基本命令检测（但**必须告知用户这是降级检测**）：

```
并行执行（每个子 agent 检测一个工具）：
  Agent: java -version
  Agent: node --version
  Agent: mvn -v（或检查 wrapper）
  ...
返回格式：{ tool, required, actual, status }
```

同时向用户说明：
```
【降级检测提示】
当前 tk-dev-env-manager 不可用，仅执行了基本的工具可达性检测。
未做版本校验和项目配置读取，可能存在版本不匹配的风险。
```

### 质量门禁 T2

- [ ] 核心工具链已检测（含版本信息）
- [ ] 环境状态已判定（区分：就绪/部分就绪/缺失）
- [ ] 缺失且影响任务的项已通过 tk-dev-env-manager 配置或用户确认跳过
- [ ] 当前任务所需环境确认可用

**未通过时的处理**：阻塞进入 T3，等待用户完成环境配置。用户明确说"跳过环境检查"时可继续，但必须在交接包中标注环境风险。

## T3: Skill 路由分发

**目的**：根据任务类型和项目技术栈，选择正确的下游 skill。

### T3.1 路由决策表

| 任务类型 | 项目技术栈 | 下游 Skill |
|----------|-----------|-----------|
| 前端开发 | Vue 3 | `tk-vue3-dev-standards` |
| 后端开发 | Spring Boot | `tk-springboot-dev-standards` |
| 数据库设计 | 任意 | `tk-database-design-standards` |
| 全栈开发 | Vue 3 + Spring Boot | 多 skill 顺序协作（见 T3.2） |
| 全栈+DB | Vue 3 + Spring Boot | 三个 skill 顺序协作（见 T3.2） |
| 环境相关 | 任意 | `tk-dev-env-manager` |
| 前端开发 | 非 Vue 3 | 通用开发流程（见 T3.3） |
| 后端开发 | 非 Spring Boot | 通用开发流程（见 T3.3） |

### T3.2 多 Skill 协作策略（全栈任务）

全栈任务涉及前后端两个 skill，协作规则：

**始终串行，先后端后前端**：
1. 如果涉及数据库设计，**先交接给 `tk-database-design-standards`** 完成表结构设计
2. 再将任务（含交接包和 DB 设计结果）交接给 `tk-springboot-dev-standards` 完成后端
3. 后端完成后，将后端产出（API 接口契约）连同前置交接包交接给 `tk-vue3-dev-standards`
4. 多个 skill 不能并行执行编码——每个阶段的产出是下一阶段的输入

**协作顺序**：
```
数据库设计 → 后端实现 → 前端实现
（每步完成后才交接给下一步）
```

**数据契约先行**：
- 全栈任务先与用户确认 API 接口契约（路径、参数、响应格式）
- 契约写入交接包，作为两个 skill 之间的"合同"

**为什么不能并行**：
- 前端页面的字段、交互逻辑依赖后端实际返回的数据结构
- 即使预先设计契约，后端实现中可能调整，前端需要同步
- 并行编码的风险远大于串行的等待成本

### T3.3 无匹配 Skill 的处理

如果项目技术栈不匹配任何已有 skill：

```
【Skill 路由结果】
当前项目技术栈：（识别到的技术栈）
未找到匹配的专用 skill。

将使用通用开发流程：按项目现有模式执行（深度阅读 → 确认方案 → 编码 → 验证）。
前置分析结果仍会打包到交接包中供参考。

请确认是否继续。
```

注意：通用流程**不是本 skill 自己编码**，而是将交接包和任务信息交给主 agent
按通用流程执行。本 skill 的职责在 T4 交接后即结束。

### T3.4 用户确认

路由决策完成后，**必须向用户展示**并等待确认。

**展示格式：**
```
【前置分析完成】
任务类型：（前端/后端/全栈/...）
任务类别：（新功能/Bug 修复/重构/...）
项目技术栈：（Vue 3 + Element Plus / Spring Boot + MyBatis / ...）
环境状态：（就绪 / 部分就绪 / 有缺失但已处理）
路由目标：（tk-vue3-dev-standards / tk-springboot-dev-standards / tk-database-design-standards / 通用流程）

即将交接给（目标）执行开发任务。
```

**确认方式：**
- 如果所有信息都已明确且无歧义，直接请用户确认是否正确：`请回复"确认"继续，或补充调整。`
- 如果有需要用户决策的点，先逐个问，收集完答案后再汇总确认
- **不要一次性列出所有问题**，而是逐个抛出，等用户回答后再提下一个

**后续处理：**
- 用户确认 → 进入 T4 交接
- 用户有调整 → 回到对应阶段修正后重新确认

### 质量门禁 T3

- [ ] 路由决策已确定
- [ ] 全栈任务的协作顺序已明确（先后端后前端）
- [ ] 用户已确认路由决策
- [ ] 无匹配 skill 时用户已确认使用通用流程（此项仅在有匹配 skill 时可跳过）

**未通过时的处理**：回到 T3.4 等待用户确认。如果用户否决路由决策，重新评估 T3.1 并再次确认。

## T4: 上下文交接

**目的**：将 T0-T3 收集的所有信息打包交接给下游 skill。

### T4.1 交接包结构

```markdown
## 前置分析交接包

### 任务信息
- 任务类别：（前端开发 / 后端开发 / 全栈开发 / Bug 修复 / 重构 / 测试 / 数据库迁移 / ...）
- 核心目标：（一句话描述）
- 原始需求：（用户的原始表述）
- 紧急度：（低 / 中 / 高）
- 需求澄清记录：（T0 中提出的问题 + 用户的答复，如无提问则标"无歧义"）

### 项目技术栈
- 项目类型：（Java Maven / Node Vite / 多模块 / ...）
- 前端框架：（Vue 3.4 + Element Plus + TypeScript / 无前端）
- 后端框架：（Spring Boot 3.2 + MyBatis-Plus / 无后端）
- 构建工具：（Vite 5 / Maven 3.9 / Gradle 8.x / ...）
- 其他关键依赖：（Redis / RabbitMQ / Lombok / ...）

### 项目结构概览
- 目录树（顶层）
- 关键目录和文件位置

### 现有资源盘点
- 前端：组件 / composables / 工具函数 / API 函数 / 类型定义 / Store
- 后端：Controller / Service / Mapper / Entity / DTO / VO / 工具类

### 环境状态
- 工具链详情：（每个工具的版本 + 状态）
- 缺失项处理：（已配置 / 已记录 / 用户跳过 / 环境风险警告）

### 路由信息
- 目标 skill：（skill 名称 / 通用流程）
- 协作模式：（单 skill / 多 skill 顺序协作）
- 如委托 tk-dev-env-manager 检测：skip_e0_e1: true（已由 T1 完成项目类型和版本解析）
- 如为全栈任务：API 接口契约（已确认的接口路径、参数、响应格式）
```

### T4.2 交接原则

**交接包是参考输入，不是替代品**：

交接包为下游 skill 提供了项目概览和技术栈快照，可以**加速**下游 skill 的 P0 阶段
（快速确认而非从头扫描），但**不能替代**下游 skill 的任何质量门禁。

具体而言：
- 下游 skill 的 P0（深度阅读）是**强制且不可跳过**的——前置分析只做顶层扫描，
  不替代逐行阅读源码
- 下游 skill 的 P0 质量门禁仍然需要逐项确认——交接包是参考数据，
  不是质量门禁的自动满足依据
- 下游 skill 的用户确认点仍然需要执行——不可因为已有前置分析就跳过确认

### T4.3 交接后的行为

交接完成后，本 skill 的职责结束。后续工作由下游 skill 全权负责。
本 skill 不应继续参与或干预下游 skill 的执行流程。

## 前置流程总览（速查）

```
触发条件：用户要求执行任何编码任务
    ↓
T0: 分析意图（主 agent，需与用户交互）
    模糊 → 提问用户 → 重新分析
    ↓
T1: 预扫描项目（⚡ 阶段内可并行读取配置文件和扫描资源目录）
    ↓
T2: 检查环境（委托 tk-dev-env-manager）
    缺失 → 引导配置 → 重新检查
    ↓
T3: 路由分发（主 agent，需与用户交互）
    用户调整 → 回退修正
    ↓
T4: 上下文交接 → 打包交接给下游 skill → 本 skill 结束
```

**关键提醒**：
- 前置流程的目的是**准备**，不是**执行**
- 子 agent 只用于**只读扫描和检测**，禁止用于编码
- 全栈任务必须**串行**（先后端后前端），不能并行编码
- 下游 skill 的 P0 是强制的，交接包是**加速参考**而非替代
