# tk-start - 任务启动与需求解析

> 本 skill 在用户提出任务时自动触发。核心职责：理解需求 → 确认意图 → 匹配 Skill → 调度执行。

---

## 第零条：铁律

1. **每次只问用户一个问题** — 不要一次抛出多个问题
2. **需求不明确时不猜测** — 必须向用户确认后才能开始执行
3. **改动范围必须获用户确认** — 禁止未经确认就修改代码

---

## 一、任务解析流程

### 1.1 解析用户需求

收到用户任务后，提取以下维度：

| 维度 | 说明 |
|------|------|
| 任务目标 | 用户想要完成什么 |
| 涉及范围 | 涉及的技术栈/模块 |
| 约束条件 | 已知的限制 |
| 期望结果 | 用户期望的交付物 |

### 1.2 展示解析结果

```
## 需求解析

**目标**：[一句话概括]
**范围**：[涉及的技术栈/模块]
**约束**：[已知的限制]
**交付物**：[期望结果]

**请确认是否正确？如有偏差请指出。**
```

### 1.3 不明确时必须提问

以下情况必须主动询问：
- 任务范围不清晰
- 存在多种实现路径
- 缺少必要上下文
- 约束条件不明确

**提问方式**：使用选项列表，提供"其他"选项让用户补充。

## 二、Skill 匹配

### 2.1 匹配规则

| 用户需求关键词 | 推荐匹配 Skill |
|---------------|---------------|
| Spring Boot / Java / 后端 / API | tk-springboot-develop-standards |
| Vue / 前端 / 组件 / TypeScript | tk-vue3-dev-standards |
| 多模块 / 前后端同时涉及 | 同时匹配多个 Skill |
| 无法匹配任何 Skill | 直接使用通用能力处理 |

### 2.2 匹配后确认

```
## Skill 匹配

根据任务内容，将使用以下规约：
- **[Skill 名称]**：[匹配原因]

**是否开始执行？**
```

---

## 三、执行前确认

### 3.1 确认检查清单

开始执行前必须确认：

- [ ] 需求已理解且用户已确认
- [ ] Skill 已匹配
- [ ] 改动范围已明确并获用户同意

### 3.2 确认原则

| 任务复杂度 | 确认次数 |
|-----------|---------|
| 简单（单文件小改动） | 1 次（执行前） |
| 中等（多文件修改） | 2 次（方案 + 执行前） |
| 复杂（多模块/新功能） | 多次（每阶段确认） |

---

## 四、执行后审视

### 4.1 交付物检查

任务完成后，检查以下内容：

| 检查项 | 说明 |
|--------|------|
| 需求匹配 | 交付物是否解决了用户提出的问题 |
| 功能完整 | 是否包含所有要求的功能点 |
| 编译验证 | 代码是否通过了编译检查 |
| 交付物齐全 | 文档/测试用例是否已输出 |

### 4.2 结果汇报

```
## 任务完成

**完成内容**：[简要说明]
**交付物**：[列出文件/文档]
**编译状态**：[通过/未通过]

**是否需要其他操作？**
```

## 五、多 Agent 协作指南

### 5.1 何时使用多 Agent

| 场景 | 是否适合 |
|------|---------|
| 多个独立模块并行开发 | 适合 |
| 前后端同时修改 | 适合 |
| 需要同时调研多个主题 | 适合 |
| 强依赖的串行任务 | 不适合 |
| 简单小任务 | 不适合 |

### 5.2 拆分原则

- 各子任务相互独立，无循环依赖
- 子任务复杂度大致均衡
- 明确各 Agent 的职责边界
