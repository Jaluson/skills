# tk-start - 任务启动与需求解析 Skill（调度器）

## 触发条件

当用户提出任何任务、问题或需求时，自动触发此 skill。

## 调度器核心原则（强制）

### 角色定位

**tk-start 是任务调度器，不是执行器。**

| 角色 | 职责 |
|------|------|
| **tk-start（调度器）** | 解析需求 → 识别类型 → 匹配 Skill → 调度分发 |
| **下游 Skill（执行器）** | 接收任务 → 执行落地 → 结果反馈 |

### 禁止行为

**绝对禁止**：
1. ❌ 不执行任何代码编写
2. ❌ 不直接运行命令或脚本
3. ❌ 不修改任何项目文件
4. ❌ 不进行编译构建
5. ❌ 不执行测试操作

**调度器只做**：
1. ✅ 解析用户需求
2. ✅ 识别任务类型
3. ✅ 匹配合适的 Skill
4. ✅ 准备任务上下文
5. ✅ 调度分发到下游 Skill

### 调度流程

```
用户提问
    │
    ▼
┌─────────────────────────────────────┐
│  解析需求（调度器职责）              │
│  - 拆解任务目标                      │
│  - 识别任务类型                       │
│  - 确定涉及范围                       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  匹配 Skill（调度器职责）            │
│  - 扫描可用 Skill                    │
│  - 选择最佳匹配                       │
│  - 检查前置条件                       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  调度分发（调度器职责）              │
│  - 准备任务上下文                    │
│  - 调用下游 Skill                    │
│  - 监控执行状态                       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  结果汇总（调度器职责）              │
│  - 收集执行结果                       │
│  - 生成任务文档                       │
│  - 向用户汇报                         │
└─────────────────────────────────────┘
```

### 前置规约

下游 Skill 执行前必须满足：

| 前置条件 | 说明 |
|----------|------|
| **需求已确认** | 用户已确认解析后的任务目标 |
| **Skill 已匹配** | 找到合适的执行 Skill |
| **环境已就绪** | 运行时环境检查通过 |
| **改动已确认** | 最小改动范围已获用户确认 |

**未满足前置条件时，禁止调度，必须先完成前置条件。**

## 核心功能

### 0. 运行时环境检查（前置条件）

**触发条件**：当任务需要运行代码时，必须先完成运行时环境检查。

#### 环境文件位置
- 项目根目录 `/runtime-env/.env` - 环境变量配置
- 项目根目录 `/runtime-env/.description` - 环境描述说明

#### 检查流程

```
任务需要运行代码
    │
    ▼
┌─────────────────────────────────────┐
│ 检查 /runtime-env/ 目录是否存在     │
└─────────────────────────────────────┘
    │
    ├── 存在 ──► 加载 .env 和 .description
    │                 │
    │                 ▼
    │      ┌─────────────────────────┐
    │      │ 检测当前项目所需环境     │
    │      └─────────────────────────┘
    │                 │
    │                 ▼
    │      ┌─────────────────────────┐
    │      │ 环境是否匹配？           │
    │      └─────────────────────────┘
    │           │是            │否
    │           ▼              ▼
    │    自动填充缺失项    提示用户配置
    │           │              │
    │           └──────┬───────┘
    │                  ▼
    │         等待用户输入/确认
    │
└── 不存在 ──► 自动创建目录和文件
                  │
                  ▼
         提示用户需要配置环境
```

#### 文件格式规范

**.env 文件格式**：
```
# 运行时环境配置
RUNTIME_NAME=项目名称
RUNTIME_TYPE=java/nodejs/python/go 等
RUNTIME_VERSION=版本号
RUNTIME_HOME=环境根目录路径
# 其他自定义环境变量
```

**.description 文件格式**：
```
# 环境描述
这是一个 [项目类型] 项目
- 主要语言：[语言]
- 框架：[框架名称]
- 数据库：[数据库类型](可选)
- Node版本：[版本](可选)
- JDK版本：[版本](可选)
- 其他依赖：[说明]

## 环境状态
- 状态：已配置/未配置/部分配置
- 最后更新：YYYY-MM-DD
```

#### 匹配检测逻辑

| 检测项 | 说明 |
|--------|------|
| **语言匹配** | 检测 .env 中的 RUNTIME_TYPE 是否与项目语言一致 |
| **版本匹配** | 检测 RUNTIME_VERSION 是否在项目支持的版本范围内 |
| **路径有效** | 检测 RUNTIME_HOME 指向的目录是否存在 |
| **依赖完整** | 检测环境是否包含项目所需的所有依赖 |

### 1. 命令执行环境兼容性规则（强制）

**核心原则**：禁止使用 Python、Perl、Ruby、PHP 等个性化脚本语言，必须优先考虑用户环境的兼容性。

#### 环境类型判断

| 环境类型 | 特征 | 可用命令 |
|----------|------|----------|
| **Windows 原生** | Windows 系统，无 WSL/Git Bash | `cmd.exe` 命令、PowerShell 基础命令 |
| **Windows + Unix 工具** | 安装了 Git Bash/MSYS/WSL | `cmd.exe` + Unix 基础命令（ls, cat, grep 等） |
| **Linux 原生** | 纯 Linux 环境 | 标准 Unix 命令 |
| **macOS** | macOS 环境 | 标准 Unix 命令 + BSD 特有命令 |

#### 禁用脚本语言清单

以下脚本语言**禁止直接调用**，除非用户明确确认环境支持：

| 禁用语言 | 原因 |
|----------|------|
| Python | 用户可能未安装 |
| Perl | 跨平台一致性差 |
| Ruby | 用户可能未安装 |
| PHP | 通常用于 Web，非通用脚本 |
| Bash/Shell | Windows 原生环境不支持 |
| PowerShell Core | 非 Windows 预装 |

#### 命令优先级规则

**必须按以下优先级选择命令**：

```
优先级 1：原生可执行命令
    └── 如：node, java, go, rustc 等项目相关运行时

优先级 2：跨平台标准命令（无外部依赖）
    ├── 文件操作：mkdir, rm, cp, mv
    ├── 文本处理：echo, type/cat, find/grep
    └── 路径操作：cd, pwd, dirname, basename

优先级 3：平台适配命令（需检测环境）
    ├── Windows：where, dir, certutil
    └── Unix：which, ls, chmod

优先级 4：项目内置工具
    └── 如：npm, mvn, gradle, dotnet, cargo
```

```
需要执行命令时
    │
    ▼
┌─────────────────────────────────────┐
│  读取 /runtime-env/.env 配置        │
│  读取 /runtime-env/.description     │

    │
    ▼
┌─────────────────────────────────────┐
│  判断任务需要的命令类型              │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  检查用户环境是否支持               │
└─────────────────────────────────────┘
    │
    ├── 支持 ──► 直接使用
    │
    └── 不支持 ──► 寻找替代方案
                      │

              └─────────────────┘
                 │是        │否

```

#### 外部命令最小化原则

**必须遵循**：
1. 能用内置工具完成就不用外部命令
2. 能用项目构建工具完成就不用额外脚本
3. 能读取配置文件就不用执行命令获取信息
4. 能复用已有工具就不用新建脚本

**判断标准**：
```
是否需要外部命令？
    │
    ├── 任务要求运行代码？
    │     └── 检查 /runtime-env/.env 中的 RUNTIME_TYPE
    │
    ├── 任务需要信息查询？
    │     └── 优先读取文件 > 执行命令
    │
    └── 任务需要转换处理？
          └── 优先用内置工具 > 用脚本
```

### 2. 拆解用户提问

将用户的原始提问分解为以下几个维度：

| 维度 | 说明 |
|------|------|
| **任务目标** | 用户想要完成什么 |
| **涉及范围** | 任务涉及的技术栈/领域/模块 |
| **约束条件** | 已知的时间、技术、环境等限制 |
| **期望结果** | 用户期望的交付物或输出 |

### 3. 展示 AI 解析结果

以结构化方式展示解析结果：

```
## 📋 需求解析结果

### 🎯 任务目标
[解析出的目标]

### 🗂️ 涉及范围
- 范围1
- 范围2

### ⚠️ 约束条件
[已知的限制条件]

### ✅ 期望结果
[用户期望的交付物]
```

### 4. 不明确之处必须向用户提问

当解析结果存在以下情况时，必须主动询问：

- 任务范围不清晰
- 存在多种可能的实现路径
- 缺少必要的上下文信息
- 约束条件不明确

**提问要求**：
- 使用菜单项格式展示选项
- 选项应简洁明了，便于用户选择
- 提供"其他"选项让用户补充说明

### ⚠️ 重要规则：每次只问一个问题

**核心原则**：一次只能向用户提问一个问题，多个问题必须按顺序逐一回答。

**为什么需要这个规则**：
- 避免用户面对多个问题感到压力
- 确保每个回答都能被充分理解和处理
- 避免回答之间的相互影响
- 提高沟通的清晰度和准确性

**具体要求**：
1. **每次只提一个问题**：即使需要多个信息，也必须逐个提问
2. **等待用户回答后再提下一个问题**：不要一次抛出多个问题
3. **基于前一个回答再提下一个问题**：根据用户的回答调整后续问题

**正确示例**：
```
## ❓ 需要确认

请先选择任务类型：

1. [ ] 开发新功能
2. [ ] 修复 Bug
3. [ ] 重构代码
4. [ ] 其他

请输入选项编号：
```
（等待用户回答后，再问下一个问题）

**错误示例**：
```
## ❓ 需要确认

请选择任务类型：1/2/3/4
请选择技术栈：Java/Go/Python
请选择优先级：高/中/低

请依次回答：
```

### 5. 用户回答后的处理流程

1. **重新理解**：基于用户的新输入，重新解析需求
2. **出具整理结果**：更新解析结果
3. **重新展示**：向用户展示最终确认版本
4. **请求确认**：明确询问用户是否确认

## 任务审视与验证（调度器职责）

### 核心原则

**调度器负责审视下游 Skill 的执行结果，确保满足需求。调度器本身不执行任务。**

### 为什么需要这个规则

- **确保质量**：下游执行可能有遗漏，需要审视
- **满足需求**：用户需求是最终衡量标准
- **及时止损**：发现问题及时让下游返工
- **完整交付**：确保文档、交付物完整

### 审视时机

```
下游 Skill 执行完成后
    │
    ▼
┌─────────────────────────────────────┐
│  调度器：审视执行结果                 │
│  （不下场执行，只审视）              │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  是否满足需求？                      │
└─────────────────────────────────────┘
```

### 审视检查点

下游 Skill 交付后，调度器逐一检查：

| 检查项 | 说明 | 执行方 |
|--------|------|--------|
| **✅ 需求匹配** | 交付物是否解决了用户提出的问题 | 调度器审视 |
| **✅ 功能完整** | 是否包含所有要求的功能点 | 调度器审视 |
| **✅ 范围覆盖** | 是否在约定的范围内工作 | 调度器审视 |
| **✅ 质量达标** | 代码是否可读、可维护、无明显错误 | 下游 Skill |
| **✅ 可正常使用** | 运行结果是否符合预期 | 下游 Skill |
| **✅ 无副作用** | 是否引入了新的问题 | 下游 Skill |

### 异常检测规则（强制）

**调度器检测下游执行结果的异常，不自行执行检测：**

#### 编译/构建异常
```
调度器检查下游报告：
- 下游是否报告编译错误？
- 下游是否报告构建失败？
- 下游是否报告资源缺失？
```

#### 运行时异常
```
调度器检查下游报告：
- 下游是否报告运行时错误？
- 下游是否报告功能异常？
- 下游是否报告错误日志？
```

#### 回归异常
```
必须检查：
1. 原有功能是否受损
2. 已有测试是否通过
3. 配置是否被意外修改
```

#### 异常分类体系

```
┌─────────────────────────────────────────┐
│              异常分类                    │
├─────────────────┬───────────────────────┤
│ 🔁 可重试       │ 网络超时、资源暂时不可用 │
│ 🔧 不可重试     │ 配置错误、权限不足、逻辑错误 │
│ ❓ 需用户决策   │ 依赖冲突、数据风险、范围变更 │
└─────────────────┴───────────────────────┘
```

#### 异常分类判断规则

| 异常特征 | 分类 | 处理方式 |
|----------|------|----------|
| 网络超时、连接重置 | 🔁 可重试 | 自动重试 |
| 资源暂时不可用 | 🔁 可重试 | 等待后重试 |
| 语法错误、类型错误 | 🔧 不可重试 | 需修复代码 |
| 权限不足 | 🔧 不可重试 | 需用户授权 |
| 依赖版本冲突 | ❓ 需用户决策 | 询问用户 |
| 数据损坏风险 | ❓ 需用户决策 | 询问用户 |
| 需求范围变更 | ❓ 需用户决策 | 询问用户 |

#### 自动重试策略配置

```
重试配置：
- 最大重试次数：3
- 重试间隔：30 秒（指数退避 30s, 60s, 120s）
- 重试超时：5 分钟

重试流程：
1. 首次失败 → 等待 30 秒 → 重试
2. 第二次失败 → 等待 60 秒 → 重试
3. 第三次失败 → 等待 120 秒 → 重试
4. 仍失败 → 标记为不可重试，报告用户
```

## 最小改动原则（强制）

### 核心原则

**必须只修改任务必要的文件，避免改动无关代码，减少影响面。**

### 为什么需要这个原则

- **降低风险**：改动越少，引入新问题的可能性越小
- **便于回滚**：当出现问题时，快速定位和回退
- **易于审查**：代码审查者可以聚焦在变更上
- **保持历史**：不破坏 git 历史和 blame 信息

### 改动范围判定

#### ✅ 必须改动的文件
```
1. 任务明确要求的文件
2. 修复 bug 必须修改的文件
3. 实现功能必须修改的文件
4. 更新版本依赖的配置文件
```

#### ❌ 禁止改动的文件
```
1. 任务范围之外的文件
2. 没有实际变更的文件（禁止无效空白修改）
3. 无关的配置文件
4. 无关的测试文件（除非任务要求）
5. 无关的文档文件（除非任务要求）
```

#### ⚠️ 谨慎改动的文件
```
1. 基类/父类（可能影响子类）
2. 接口定义（可能影响实现方）
3. 公共工具类（可能影响调用方）
4. 配置文件（确认变更必要性）

## Skill 调度规范

### 调度原则

**tk-start 作为调度器，必须将任务交给下游 Skill 执行：**

| 原则 | 说明 |
|------|------|
| **只调度不执行** | tk-start 绝不直接执行任务，只负责调度 |
| **匹配最优 Skill** | 根据任务类型匹配最合适的下游 Skill |
| **传递完整上下文** | 将解析结果、用户确认、环境信息完整传递给下游 |
| **监控不替代** | 监控下游执行状态，但不介入执行本身 |

### Skill 匹配时机

```
任务确认后，立即进行 Skill 匹配：

1. 任务类型识别后
2. 前置条件确认后
3. 用户明确授权调度时
```

## 上下文管理规范

### 基本原则

在调度任务过程中，必须平衡上下文大小与任务质量：

| 原则 | 说明 |
|------|------|
| **相关性** | 只保留与调度相关的信息，传递给下游的需精简 |
| **层次性** | 摘要提炼重要信息，避免堆砌原始数据 |
| **连续性** | 关键决策、用户确认、依赖关系必须保留 |
| **隔离性** | 下游执行细节不需全部保留，保留结论即可 |

### 自动压缩时机

```
触发条件（满足任一即触发）：
- 对话轮次超过 15 轮
- 上下文使用量超过 60%
- 单次 Agent 任务完成时

压缩操作：
1. 将已完成任务的详细日志转为摘要
2. 保留关键决策点、用户确认、依赖关系
3. 清理已废弃的临时文件记录
```

### 动态上下文压缩策略

#### 压缩时机触发器

| 触发条件 | 压缩比例 | 说明 |
|----------|----------|------|
| 轮次 > 15 | 20% | 基础压缩 |
| 上下文 > 60% | 30% | 轻度压缩 |
| 轮次 > 25 | 40% | 中度压缩 |
| 上下文 > 80% | 50% | 深度压缩 |
| 任务阶段完成 | 25% | 自动压缩 |

#### 基于任务阶段的压缩策略

```
任务阶段：
├── 启动阶段（0-10%）
│   └── 保留全部原始信息
├── 执行阶段（10-80%）
│   └── 按轮次触发压缩
└── 完成阶段（80-100%）
    └── 保留结论，清理过程
```

#### 用户交互频率影响

```
高交互频率（用户频繁回复）：
- 降低压缩触发阈值
- 保留更多对话历史

低交互频率（用户偶尔回复）：
- 提高压缩触发阈值
- 更激进地压缩
```

## 任务文档输出规则
