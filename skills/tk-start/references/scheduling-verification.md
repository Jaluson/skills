# 调度验证方法论

## 概述

本文档定义 tk-start 调度器的验证机制，确保调度逻辑正确性、任务执行有效性和系统稳定性。

## 调度逻辑验证

### Skill 匹配算法测试

#### 测试用例设计

| 测试场景 | 输入 | 期望输出 | 验证点 |
|----------|------|----------|--------|
| 精确匹配 | "Java Spring Boot API 开发" | tk-springboot-dev | 类型+技术栈+关键词 |
| 技术栈优先 | "Python 数据分析" | tk-python-data | 技术栈权重最高 |
| 多技能组合 | "Go 微服务开发 + 测试" | tk-go-dev + tk-testing | 多 Skill 组合 |
| 无匹配 | "汇编语言开发" | 提示无合适 Skill | 降级处理 |
| 部分匹配 | "代码开发" | 候选 Skill 列表 | 模糊匹配 |

#### 匹配算法验证流程

```
测试输入
    │
    ▼
┌─────────────────────────────────────┐
│  特征提取验证                        │
│  - 关键词提取正确性                  │
│  - 技术栈识别准确性                  │
│  - 领域分类合理性                   │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  权重计算验证                        │
│  - 各维度权重分配                    │
│  - 综合得分计算                      │
│  - 排序正确性                       │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  前置条件验证                        │
│  - 环境要求检查                      │
│  - 依赖可用性                        │
│  - Skill 可用性                      │
└─────────────────────────────────────┘
    │
    ▼
    测试结果
```

#### 匹配测试示例

```python
# 测试用例：Skill 匹配算法
def test_skill_matching():
    test_cases = [
        {
            "input": "开发一个 Java Spring Boot 用户管理 API",
            "expected": "tk-springboot-dev",
            "threshold": 0.6
        },
        {
            "input": "修复数据库连接问题",
            "expected": "tk-database-debug",
            "threshold": 0.5
        },
        {
            "input": "优化前端性能",
            "expected": "tk-vue3-dev 或 tk-react-dev",
            "threshold": 0.4
        }
    ]
    
    for case in test_cases:
        result = match_skill(case["input"])
        assert result.score >= case["threshold"]
        assert case["expected"] in result.skill_name
```

### 任务拆解正确性验证

#### 拆解质量检查点

```
任务拆解完成后，必须验证：
1. 完整性：所有子任务覆盖原始需求
2. 独立性：子任务间无循环依赖
3. 可执行性：每个子任务可独立执行
4. 均衡性：子任务复杂度大致相当
```

#### 拆解验证流程

```
任务拆解结果
    │
    ▼
┌─────────────────────────────────────┐
│  完整性检查                          │
│  - 原始需求覆盖率                    │
│  - 关键功能点覆盖                    │
│  - 边界条件覆盖                      │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  依赖分析                            │
│  - 构建依赖图                        │
│  - 检测循环依赖                      │
│  - 计算关键路径                      │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  均衡性分析                          │
│  - 估算各子任务复杂度                │
│  - 检测异常值                        │
│  - 建议调整                          │
└─────────────────────────────────────┘
    │
    ▼
    拆解评估报告
```

#### 拆解验证示例

```
原始需求：用户管理系统
拆解结果：
├── Agent-1: 用户注册模块
├── Agent-2: 用户登录模块
├── Agent-3: 用户信息管理
└── Agent-4: 权限控制

验证结果：
├── 完整性：✅ 覆盖 95%

├── 可执行性：✅ 全部可执行
└── 均衡性：⚠️ Agent-4 复杂度较高，建议拆分
```

### 前置条件检查验证

#### 前置条件检查清单

```
调度前必须验证：
1. ✅ 需求已确认（用户确认或默认确认）
2. ✅ Skill 已匹配（权重 >= 0.6 或用户确认）
3. ✅ 环境已就绪（运行时环境检查通过）
4. ✅ 改动已确认（最小改动范围获用户确认）
5. ✅ 权限已获取（如需敏感操作）
```

#### 前置条件检查流程

```
前置条件检查
    │
    ▼
┌─────────────────────────────────────┐
│  需求确认状态                        │
│  - 用户已明确确认？                  │
│  - 默认确认超时？（> 5 分钟无异议）  │
└─────────────────────────────────────┘
    │
    ├── 未确认 ──► 阻塞调度
    │
    ▼
┌─────────────────────────────────────┐
│  Skill 匹配状态                      │
│  - 有匹配结果？                      │
│  - 匹配权重 >= 阈值？               │
│  - 用户确认使用？                    │
└─────────────────────────────────────┘
    │
    ├── 未匹配/未确认 ──► 阻塞调度
    │
    ▼
┌─────────────────────────────────────┐
│  环境就绪状态                        │
│  - runtime-env 存在？               │
│  - 环境配置完整？                    │
│  - 依赖项可用？                      │
└─────────────────────────────────────┘
    │
    ├── 未就绪 ──► 阻塞调度
    │
    ▼
┌─────────────────────────────────────┐
│  改动确认状态                        │
│  - 改动范围已明确？                  │
│  - 用户已确认改动？                  │
└─────────────────────────────────────┘
    │
    └── 全部通过 ──► 执行调度
```

## 调度流程端到端测试

### 测试场景设计

| 场景 | 描述 | 预期结果 |
|------|------|----------|
| 正常流程 | 完整任务从解析到完成 | 任务成功完成 |
| 确认中断 | 用户在确认阶段取消 | 任务终止，无副作用 |
| Skill 无匹配 | 任务无法匹配任何 Skill | 提示用户，无异常 |
| 环境异常 | 运行时环境检查失败 | 阻塞调度，提示配置 |
| 并行调度 | 多 Agent 并行执行 | 全部完成或按策略处理 |
| 部分失败 | 多 Agent 中部分失败 | 按策略处理（继续/终止） |
| 超时处理 | 任务执行超时 | 触发超时处理流程 |

### 端到端测试流程

```
开始测试
    │
    ▼
┌─────────────────────────────────────┐
│  步骤 1：任务解析                    │
│  - 输入：用户原始需求                │
│  - 验证：解析结果结构正确            │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  步骤 2：用户确认                    │
│  - 输入：解析结果                    │
│  - 验证：确认流程正常                │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  步骤 3：Skill 匹配                  │
│  - 输入：确认后的需求                │
│  - 验证：匹配结果合理                │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  步骤 4：环境检查                    │
│  - 输入：运行时环境                  │
│  - 验证：检查结果准确                │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  步骤 5：调度执行                    │
│  - 输入：完整的上下文                │
│  - 验证：调度分发正确                │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  步骤 6：结果汇总                    │
│  - 输入：下游执行结果                │
│  - 验证：结果汇总完整                │
└─────────────────────────────────────┘
    │
    ▼
    测试完成
```

### 回归测试

```
回归测试触发条件：
- 调度器代码变更后
- Skill 配置变更后
- 新增 Skill 后

回归测试内容：
1. 核心调度流程不受影响
2. Skill 匹配结果一致
3. 异常处理流程正常
4. 用户交互正常
```

## 多 Agent 协作测试

### 并行任务执行验证

```
并行执行测试点：
1. 任务分配均衡性
2. 资源竞争处理
3. 结果汇总正确性
4. 性能提升验证
```

### 依赖关系处理验证

```
依赖处理测试场景：

场景 1：顺序依赖
A ──► B ──► C
验证：B 在 A 完成后启动

场景 2：并行依赖
A ──► B
C ──► B
验证：B 在 A 和 C 都完成后启动

场景 3：条件依赖
A ──► B
A ──► C
验证：根据 A 结果决定 B/C 执行

场景 4：循环依赖检测
A ──► B
B ──► C
C ──► A
验证：检测并报告循环依赖错误
```

### 状态同步验证

```
状态同步测试：
1. 初始状态正确设置
2. 状态转换正确触发
3. 状态通知及时送达
4. 最终状态一致
```

### 异常传播验证

```
异常传播测试场景：

场景：Agent-2 发生异常
预期：
- Agent-2 状态变为异常
- 依赖 Agent-2 的 Agent-3 等待
- 主线程收到异常通知
- 用户收到异常报告
- 可选择重试/跳过/终止
```
