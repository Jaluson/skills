# 任务拆解方法论

## 拆解原则

### 1. MECE 原则
- **相互独立**：各子任务不重叠
- **完全穷尽**：覆盖所有必要方面

### 2. 渐进式拆解
先大后小，先粗后细

### 3. 可执行性原则
每个子任务应可独立执行和验收

## 拆解维度

### 目标维度
- 最终交付物是什么？
- 成功的衡量标准是什么？
- 需要达到的质量水平？

### 范围维度
- 涉及哪些技术栈？
- 涉及哪些模块/组件？
- 是否有边界限制？

### 资源维度
- 可用的技术资源？
- 时间限制？
- 人员限制？

### 风险维度
- 已知的技术难点？
- 可能的风险点？
- 需要的外部依赖？

### 编译验证强制执行流程

```
任务完成
    │
    ▼

│  - 代码语法检查                      │
│  - 类型检查（如适用）                │
│  - 依赖解析                          │
└─────────────────────────────────────┘
    │
    ├── 编译失败 ──► 返回修复 ─────────┐
    │                                  │
    └── 编译成功 ──► 继续验证          │
    │                                  │
    ▼                                  │
┌─────────────────────────────────────┐
│  单元测试阶段                        │
│  - 新增代码测试                      │
│  - 回归测试                          │
└─────────────────────────────────────┘
    │
    ├── 测试失败 ──► 返回修复 ─────────┐
    │                                  │
    └── 测试成功 ──► 验证完成          │
    │                                  │
    └──────────────────────────────────┘
```

### 回归检测机制

```
回归检测清单：
1. 原有功能是否受损
2. 已有测试是否通过
3. 配置是否被意外修改
4. API 接口是否保持兼容
5. 性能是否下降

回归测试执行：
- 运行完整测试套件
- 对关键路径进行冒烟测试
- 比对性能基准
```

## 问题发现清单

当遇到以下情况时，必须标记为"需确认"：

- [ ] 任务目标模糊或有多重含义
- [ ] 范围边界不清晰
- [ ] 缺少必要的上下文
- [ ] 存在歧义的术语
- [ ] 时间要求不明确
- [ ] 优先级不明确
- [ ] 质量标准不明确
- [ ] 假设条件未验证

## 拆解质量评估

### 拆解评分矩阵

| 维度 | 优秀 (4) | 良好 (3) | 一般 (2) | 差 (1) |
|------|----------|----------|----------|---------|
| 完整性 | 无遗漏 | 轻微遗漏 | 部分遗漏 | 严重遗漏 |
| 独立性 | 完全独立 | 少量依赖 | 中度依赖 | 强耦合 |
| 可执行性 | 明确可执行 | 基本可执行 | 需要澄清 | 难以执行 |
| 均衡性 | 完全均衡 | 基本均衡 | 部分不均 | 严重不均 |

### 拆解评审检查点

```
拆解完成后，评审以下内容：
1. 是否覆盖了所有必要的子任务？
2. 子任务之间是否有不必要的重叠？
3. 子任务是否可以独立验收？
4. 各子任务的复杂度是否大致均衡？
5. 依赖关系是否清晰明确？
6. 是否有遗漏的关键路径？
```
