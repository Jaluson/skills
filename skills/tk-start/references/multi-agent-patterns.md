# 多 Agent 协作模式参考

## 何时使用多 Agent

### 适合场景

| 场景 | 特征 | 收益 |
|------|------|------|
| **并行模块开发** | 多个独立功能模块 | 耗时减少约 60% |
| **多技术栈任务** | 前端/后端/数据库/运维 | 分工明确，减少上下文切换 |
| **大规模重构** | 多个文件/组件同时修改 | 并行处理提高效率 |
| **调研任务** | 需要同时研究多个主题 | 节省 50%+ 时间 |
| **测试生成** | 多个模块同时编写测试 | 覆盖更全面 |

### 不适合场景

| 场景 | 原因 |
|------|------|
| **强依赖任务** | 子任务必须串行执行 |
| **简单短任务** | Agent 启动开销大于任务本身 |
| **需要统一决策** | 多 Agent 可能产生不一致 |

## Agent 生命周期管理

### 生命周期状态机

```
┌─────────────────────────────────────────────────────────────┐
│                      Agent 生命周期                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│    ┌──────────┐                                             │
│    │  初始化   │                                             │
│    └────┬─────┘                                             │
│         │                                                    │
│         ▼                                                    │
│    ┌──────────┐     ┌──────────┐     ┌──────────┐           │
│    │  就绪    │────▶│  运行中   │────▶│  完成    │           │
│    └──────────┘     └────┬─────┘     └──────────┘           │
│         ▲                │                                   │
│         │                ▼                                   │
│         │           ┌──────────┐                             │
│         │           │  异常    │                             │
│         │           └────┬─────┘                             │
│         │                │                                   │
│         │                ▼                                   │
│         │           ┌──────────┐                             │
│         └──────────│  等待中   │                             │
│                     └──────────┘                             │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 各状态说明

| 状态 | 进入条件 | 退出条件 | 可执行操作 |
|------|----------|----------|-----------|
| 初始化 | 创建 Agent 实例 | 初始化完成 | 配置参数 |
| 就绪 | 初始化完成/等待依赖 | 收到启动信号 | 等待 |
| 运行中 | 收到启动信号 | 任务完成/异常/停止 | 执行任务 |
| 异常 | 执行出错 | 重试/用户决策/放弃 | 等待处理 |
| 完成 | 任务成功完成 | - | 输出结果 |
| 等待中 | 依赖未满足/优雅停止 | 依赖满足/超时 | 等待 |

### 生命周期转换触发器

```
初始化 → 就绪：
    - Skill 上下文准备完成
    - 依赖项已声明

就绪 → 运行中：
    - 所有依赖 Agent 已完成
    - 用户明确启动

运行中 → 完成：
    - 任务执行成功
    - 交付物已产出

运行中 → 异常：
    - 执行出错
    - 超时

运行中 → 等待中：
    - 收到停止信号
    - 依赖项状态变更

异常 → 就绪：
    - 重试成功
    - 用户决定重试

异常 → 完成：
    - 跳过此任务
    - 用户决定放弃
```

## 任务依赖关系图

### 依赖图表示法

```
依赖图结构：
{
  "nodes": [
    { "id": "A", "name": "数据模型设计", "agent": "Agent-1" },
    { "id": "B", "name": "API 开发", "agent": "Agent-2" },
    { "id": "C", "name": "前端开发", "agent": "Agent-3" },
    { "id": "D", "name": "集成测试", "agent": "Agent-4" }
  ],
  "edges": [
    { "from": "A", "to": "B", "type": "output" },
    { "from": "A", "to": "C", "type": "output" },
    { "from": "B", "to": "D", "type": "input" },
    { "from": "C", "to": "D", "type": "input" }
  ]
}
```

### 依赖类型

| 类型 | 符号 | 说明 | 调度影响 |
|------|------|------|----------|
| 输出依赖 | ──► | A 的输出是 B 的输入 | B 必须等待 A 完成 |
| 顺序依赖 | ──▶ | B 必须在 A 之后执行 | B 必须等待 A 完成 |
| 条件依赖 | ──? | B 依赖 A 的结果决定是否执行 | A 完成后判断 |
| 无依赖 | ──✗ | 两者可完全并行 | 可同时执行 |

### 依赖图解析算法

```
输入：依赖图
输出：执行序列（可并行分组）

1. 计算入度（每个节点的依赖数量）
2. 找出所有入度为 0 的节点（可立即执行）
3. 标记这些节点为当前批次
4. 从图中移除这些节点
5. 更新受影响节点的入度
6. 重复步骤 2-5 直到所有节点完成

示例：
依赖关系：
A ──► B ──► D
C ──► D

执行序列：
- 批次 1：[A, C]（可并行）
- 批次 2：[B]（等待 A）
- 批次 3：[D]（等待 B, C）
```

### 依赖关系可视化

```
横向展示：
┌─────────────────────────────────────────────────────────────┐
│  批次 1        批次 2        批次 3        批次 4         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌────────┐                                                │
│  │ Agent-A │────────────────────────────────────────┐       │
│  └────────┘                                        │       │
│       │                                              ▼       │
│       │                                        ┌────────┐   │
│       └──────────────────┐                     │ Agent-D │   │
│                          │                     └────────┘   │
│  ┌────────┐              │                          ▲       │
│  │ Agent-C │──────────────┴──────────────────────────┘       │
│  └────────┘              │                                   │
│       │                  ▼                                    │
│       │             ┌────────┐                                │
│       └─────────────▶│ Agent-B │                                │
│                     └────────┘                                │
└─────────────────────────────────────────────────────────────┘

纵向展示：
Agent-A ████████████
         └──▶ Agent-B ──────────────────▶ Agent-D
Agent-C ████████████                              ▲
         └─────────────────────────────▶──────────┘
```

## 并行度自适应调整策略

### 调整因素

| 因素 | 说明 | 权重 |
|------|------|------|
| 任务复杂度 | 子任务的预估执行时间 | 30% |
| 上下文使用量 | 当前上下文占用比例 | 25% |
| 依赖关系 | 任务间的依赖紧密度 | 20% |
| 资源可用性 | 系统资源（内存、CPU） | 15% |
| 历史表现 | 类似任务的 Agent 表现 | 10% |

### 动态调整算法

```
初始并行度 = min(可用 Agent 数, 推荐并行度)

每次事件触发时（任务完成/失败/新任务）：
    │
    ▼
┌─────────────────────────────────────┐
│  评估当前并行度                      │
│  - 计算当前负载率                    │
│  - 检查上下文剩余空间                │
│  - 评估待处理任务数                  │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  判断是否需要调整                    │
│  - 负载率 > 80% → 降低并行度        │
│  - 负载率 < 30% → 提高并行度        │
│  - 上下文 > 70% → 降低并行度        │
│  - 上下文 < 40% → 提高并行度        │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  执行调整                            │
│  - 新增 Agent（需要时启动）         │
│  - 合并任务（负载不均时）           │
│  - 拆分任务（单个过重时）           │
└─────────────────────────────────────┘
```

### 调整阈值配置

```
并行度调整阈值：
{
  "scale_up_threshold": {
    "context_usage": 40,      // 上下文使用 < 40% 时可增加
    "load_factor": 30,        // 系统负载 < 30% 时可增加
    "queue_depth": 5          // 待处理任务 > 5 时可增加
  },
  "scale_down_threshold": {
    "context_usage": 70,      // 上下文使用 > 70% 时需减少
    "load_factor": 80,        // 系统负载 > 80% 时需减少
    "queue_depth": 2          // 待处理任务 < 2 时可减少
  },
  "parallelism_range": {
    "min": 1,
    "max": 6
  }
}
```

### 自适应调整示例

```
场景：用户管理系统开发（6 个子任务）

初始状态：
- 可用资源：上下文 60% 空闲
- 预估并行度：4

执行中检测：
- Agent-1 提前完成，释放资源
- 上下文使用下降至 45%
- 待处理任务：3 个

调整决策：
- 并行度 3 → 4（符合 scale_up 条件）

新增 Agent-4 启动
```

## Agent 间消息传递机制

### 消息类型

| 消息类型 | 方向 | 用途 | 优先级 |
|----------|------|------|--------|
| 启动消息 | 主线程 → Agent | 触发开始执行 | P0 |
| 停止消息 | 主线程 → Agent | 请求优雅停止 | P0 |
| 完成消息 | Agent → 主线程 | 报告任务完成 | P0 |
| 异常消息 | Agent → 主线程 | 报告执行异常 | P0 |
| 依赖就绪 | Agent → Agent | 通知依赖已满足 | P1 |
| 数据传递 | Agent → Agent | 传递输出数据 | P1 |
| 状态同步 | Agent → 主线程 | 定期状态汇报 | P2 |
| 心跳消息 | Agent → 主线程 | 证明存活 | P2 |

### 消息格式规范

```
消息结构：
{
  "id": "msg-uuid-001",
  "type": "DEPENDENCY_READY",
  "from": "agent-1",
  "to": "agent-2",
  "payload": {
    "dependency_id": "task-a",
    "output_location": "/tmp/output-a.json"
  },
  "timestamp": "2026-04-18T10:35:22Z",
  "priority": "P1"
}
```

### 消息队列管理

```
消息处理流程：
    │
    ▼
┌─────────────────────────────────────┐
│  消息接收                            │
│  - 按接收顺序入队                    │
│  - 根据优先级排序                    │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  优先级分发                          │
│  P0 消息 → 即时处理                 │
│  P1 消息 → 当前批次处理              │
│  P2 消息 → 批量处理（每 30 秒）     │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│  消息消费                            │
│  - 目标 Agent 处理                   │
│  - 发送确认                          │
│  - 异常重试（最多 3 次）             │
└─────────────────────────────────────┘
```

### 消息超时处理

```
超时配置：
- 启动消息超时：60 秒
- 停止消息超时：30 秒
- 完成消息超时：无（等待完成）
- 依赖就绪超时：等待直到依赖完成
- 心跳超时：120 秒（无心跳则判定异常）

超时处理：
1. 记录超时事件
2. 触发重试机制
3. 超过最大重试后，标记为异常
4. 通知主线程
```

## Agent 角色定义

### 通用型 Agent
适用于大多数任务，直接执行具体工作。

### 协调型 Agent
负责分配任务、监控进度、汇总结果。

### 专家型 Agent
针对特定领域（数据库、安全、性能等）提供深度支持。

## 任务拆解原则

### 独立性和均衡性

```
好例子：
├── Agent-1: 用户认证模块（独立）
├── Agent-2: 数据展示模块（独立）
└── Agent-3: 文件上传模块（独立）

不好的例子：
├── Agent-1: 登录功能 + 用户主页（耦合）
├── Agent-2: 只写一行配置（负载不均）
└── Agent-3: 剩下所有功能（负载不均）
```

### 依赖关系管理

```
无依赖：完全并行
    Agent-1 ─┬─► 结果A
              ├─► 结果B
              └─► 结果C

顺序依赖：
    Agent-1 ──► 结果A ──► Agent-2 ──► 结果B

条件依赖：
    Agent-1 ──► 决策 ──┬─► Agent-2A
                       └─► Agent-2B
```

## 进度追踪规范

### 状态定义

| 状态 | 符号 | 说明 |
|------|------|------|
| 等待中 | ⚪ | 未开始，等待依赖 |
| 运行中 | 🟢 | 正在执行 |
| 完成 | ✅ | 成功完成 |
| 异常 | 🔴 | 执行失败 |
| 跳过 | ⏭️ | 被用户跳过 |

### 进度条规范

```
进度条组成：█ 和 ░ 的组合，共 10 个字符
进度计算：已完成步骤 / 总步骤

示例：
░░░░░░░░░░ 0%   - 未开始
████░░░░░░ 40%  - 进行中
████████░░ 80%  - 接近完成
██████████ 100% - 完成
```

## 日志级别

### 关键日志（必须展示）
- Agent 启动/完成/异常
- 子任务开始/完成
- 遇到的问题及错误信息

### 详细日志（按需展示）
- 具体的代码修改
- 中间的思考过程
- 文件操作详情

## 异常处理策略

### 自动重试
```
可重试错误（网络超时、资源暂时不可用）：
  └─ 自动重试 1 次
  └─ 等待 30 秒后重试 2 次
  └─ 报告最终结果
```

### 需要用户决策
```
不可重试错误（权限不足、配置错误）：
  └─ 暂停此 Agent
  └─ 展示错误详情
  └─ 等待用户选择：
      1. 重试
      2. 跳过
      3. 终止全部
```

### 部分失败处理
```
多个 Agent 协作时，部分失败：
  └─ 继续执行其他成功的 Agent
  └─ 汇总失败列表
  └─ 提供整体完成度报告
  └─ 用户决定下一步
```

## 性能优化

### 并行度选择
```
CPU 核心数 = 推荐并行 Agent 数

推荐：
- 2-4 核：最多 2 个 Agent
- 4-8 核：最多 3-4 个 Agent
- 8+ 核：最多 4-6 个 Agent

注意：不要超过实际需要，避免资源竞争
```

### 负载均衡
```
均衡原则：
1. 估算每个子任务的复杂度
2. 复杂任务分配给单个 Agent
3. 简单任务可以合并
4. 避免某个 Agent 过于空闲或繁忙
```

## 用户交互

### 实时更新
- 进度变化时立即更新展示
- 日志采用滚动方式，最新在上
- 异常信息用醒目的 🔴 标记

### 定期汇报
- 每完成一个子任务，汇报一次
- 整体进度每分钟汇报一次
- 有重要里程碑时主动提醒

### 中断处理
```
用户请求中断时：
1. 立即停止新任务分配
2. 等待运行中的 Agent 优雅停止
3. 汇总已完成的工作
4. 展示已交付物和待完成项
5. 等待用户下一步指示
```
