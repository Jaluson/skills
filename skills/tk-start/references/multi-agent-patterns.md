# 多 Agent 协作模式参考

## 何时使用多 Agent

### 适合场景

| 场景 | 特征 | 收益 |
|------|------|------|
| **并行模块开发** | 多个独立功能模块 | 耗时减少约 60% |
| **多技术栈任务** | 前端/后端/数据库/运维 | 分工明确，减少上下文切换 |
| **大规模重构** | 多个文件/组件同时修改 | 并行处理提高效率 |
| **调研任务** | 需要同时研究多个主题 | 节省 50%+ 时间 |
| **测试生成** | 多个模块同时编写测试 | 覆盖更全面 |

### 不适合场景

| 场景 | 原因 |
|------|------|
| **强依赖任务** | 子任务必须串行执行 |
| **简单短任务** | Agent 启动开销大于任务本身 |
| **需要统一决策** | 多 Agent 可能产生不一致 |

## Agent 角色定义

### 通用型 Agent
适用于大多数任务，直接执行具体工作。

### 协调型 Agent
负责分配任务、监控进度、汇总结果。

### 专家型 Agent
针对特定领域（数据库、安全、性能等）提供深度支持。

## 任务拆解原则

### 独立性和均衡性

```
好例子：
├── Agent-1: 用户认证模块（独立）
├── Agent-2: 数据展示模块（独立）
└── Agent-3: 文件上传模块（独立）

不好的例子：
├── Agent-1: 登录功能 + 用户主页（耦合）
├── Agent-2: 只写一行配置（负载不均）
└── Agent-3: 剩下所有功能（负载不均）
```

### 依赖关系管理

```
无依赖：完全并行
    Agent-1 ─┬─► 结果A
              ├─► 结果B
              └─► 结果C

顺序依赖：
    Agent-1 ──► 结果A ──► Agent-2 ──► 结果B

条件依赖：
    Agent-1 ──► 决策 ──┬─► Agent-2A
                       └─► Agent-2B
```

## 进度追踪规范

### 状态定义

| 状态 | 符号 | 说明 |
|------|------|------|
| 等待中 | ⚪ | 未开始，等待依赖 |
| 运行中 | 🟢 | 正在执行 |
| 完成 | ✅ | 成功完成 |
| 异常 | 🔴 | 执行失败 |
| 跳过 | ⏭️ | 被用户跳过 |

### 进度条规范

```
进度条组成：█ 和 ░ 的组合，共 10 个字符
进度计算：已完成步骤 / 总步骤

示例：
░░░░░░░░░░ 0%   - 未开始
████░░░░░░ 40%  - 进行中
████████░░ 80%  - 接近完成
██████████ 100% - 完成
```

## 日志级别

### 关键日志（必须展示）
- Agent 启动/完成/异常
- 子任务开始/完成
- 遇到的问题及错误信息

### 详细日志（按需展示）
- 具体的代码修改
- 中间的思考过程
- 文件操作详情

## 异常处理策略

### 自动重试
```
可重试错误（网络超时、资源暂时不可用）：
  └─ 自动重试 1 次
  └─ 等待 30 秒后重试 2 次
  └─ 报告最终结果
```

### 需要用户决策
```
不可重试错误（权限不足、配置错误）：
  └─ 暂停此 Agent
  └─ 展示错误详情
  └─ 等待用户选择：
      1. 重试
      2. 跳过
      3. 终止全部
```

### 部分失败处理
```
多个 Agent 协作时，部分失败：
  └─ 继续执行其他成功的 Agent
  └─ 汇总失败列表
  └─ 提供整体完成度报告
  └─ 用户决定下一步
```

## 通信规范

### Agent 间通知
```
Agent-1 --[完成通知]--> Agent-2

时间：10:35:22
```

```
每 60 秒同步一次全局状态：

├─────────────────────────────────────┤

│ 运行中：2 | 等待中：1 | 完成：1      │
│ 异常：0                              │
└─────────────────────────────────────┘
```

## 性能优化

### 并行度选择
```
CPU 核心数 = 推荐并行 Agent 数

推荐：
- 2-4 核：最多 2 个 Agent
- 4-8 核：最多 3-4 个 Agent
- 8+ 核：最多 4-6 个 Agent

注意：不要超过实际需要，避免资源竞争
```

### 负载均衡
```
均衡原则：
1. 估算每个子任务的复杂度
2. 复杂任务分配给单个 Agent
3. 简单任务可以合并
4. 避免某个 Agent 过于空闲或繁忙
```

## 用户交互

### 实时更新
- 进度变化时立即更新展示
- 日志采用滚动方式，最新在上
- 异常信息用醒目的 🔴 标记

### 定期汇报
- 每完成一个子任务，汇报一次
- 整体进度每分钟汇报一次
- 有重要里程碑时主动提醒

### 中断处理
```
用户请求中断时：
1. 立即停止新任务分配
2. 等待运行中的 Agent 优雅停止
3. 汇总已完成的工作
4. 展示已交付物和待完成项
5. 等待用户下一步指示
```
