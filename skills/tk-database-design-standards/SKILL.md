# 数据库设计规范 — AI 自动化执行流水线

## 前置条件：环境检测（强制）

在执行数据库设计任务之前，**必须先完成环境检测**：

```
┌─────────────────────────────────────────────┐
│  1. 调用 tk-dev-env-manager 执行环境检测      │
│     - 检测数据库客户端是否安装                │
│     - 检测数据库连接信息是否配置               │
│  2. 环境检测通过 → 继续执行 D0 阶段           │
│  3. 环境检测未通过 → 完成环境配置后再继续      │
└─────────────────────────────────────────────┘
```

**未通过环境检测的代码不得交付。**

本 skill 是 AI 执行数据库设计任务时**必须严格遵循的执行流水线**。
每个阶段都有明确的输入、输出和质量门禁，只有通过当前门禁才能进入下一阶段。

### 三条不可违反的铁律

**铁律一：不明确就提问，禁止猜测。**
需求中任何模糊、有歧义、或可以有多种理解的表述，都必须向用户提问澄清。
例如：用户说"加个用户表"——需要确认是否有软删除、是否有租户隔离、是否需要审计字段等。

**铁律二：分析完毕必须展示理解，等待用户确认后才动手。**
每完成一个分析阶段（D0/D1/D2），必须向用户主动展示你对需求的理解、
你的发现和你的计划。用户说"可以""没问题""继续"后才能进入下一阶段。
禁止跳过确认直接开始写 DDL。

**铁律三：先读后写，遵循已有模式。**
进入已有项目时，必须先读取现有的表结构、命名规范、已有的 migration 文件，
严格遵循项目已有的模式，不引入新的设计风格。

**铁律四：提问后必须重新确认。** 向用户提问获取回答后，必须基于回答重新出具完整方案，并将该方案以菜单选项形式交给用户最终确认。用户选择确认选项后，方可进入下一步。未获得用户确认的方案不得执行。

**铁律五：方案必须包含详细的改动信息。** 提交给用户确认的方案中，对于每个文件变更必须包含：
- **新建表**：表完整名称、表用途、字段清单（含字段名、类型、约束、默认值、注释）、索引设计（含索引名、字段、类型、设计理由）
- **修改表**：表名称、修改内容（原结构 → 新结构）、数据迁移方案（如有）
- **删除表**：表名称、删除原因、对现有数据的影响

**铁律六：允许并行 Agent 加速。** 在 D0-D3 阶段中，当任务涉及多个表或多项检测时，允许启用最多 5 个子 agent 并行执行以加速任务：
- D0 阶段：多个表的现有结构扫描可以并行
- D1 阶段：多个表的方案设计可以并行
- D2 阶段：多个表的 DDL 编写可以并行
- D3 阶段：多个表的验证审查可以并行

子 agent 必须有明确的输入、输出和职责边界，只负责只读操作（扫描/检测）或指定任务的执行，不负责写操作（编码/修改文件）之外的决策。

示例格式：
```
【数据库变更清单】
新建表：
  - sys_user（用户表）                               — 存储系统用户信息
    字段：id(BIGINT,PK), username(VARCHAR(50),uk), password(VARCHAR(128)), deleted(TINYINT)
    索引：uk_username(username) — 用户名唯一约束
    关联：无外键，通过应用层保证一致性

修改表：
  - order 订单表                                       — 新增 address_id 字段
    新增字段：address_id BIGINT COMMENT '收货地址ID'
    数据迁移：无需迁移历史数据，新字段默认为 NULL

数据库配置变更：
  - 字符集：utf8mb4
  - 存储引擎：InnoDB
```

禁止仅用"加个用户表"等模糊描述，必须逐项列出具体表名、字段详情和索引设计。

特别注意：如果存在 `tk-dev-task-precheck` skill，数据库设计任务应优先通过它作为入口，
由它完成前置分析后再交接给本 skill，而非直接触发本 skill。
如果本 skill 是被 `tk-dev-task-precheck` 交接触发的，接收交接包中的预扫描结果
作为参考输入，但本 skill 的 D0 需求分析阶段仍然是强制的，不可跳过。

### 流程降级机制

标准流程是完整 5 阶段，但允许根据**任务复杂度**降级，降级不等于放弃质量：

| 降级级别 | 适用场景 | 可精简的阶段 | 不可降级的底线 |
|----------|---------|-------------|---------------|
| 完整流程 | 新建多张关联表、复杂表结构设计 | 无 | 全部 D0-D4 |
| 标准流程 | 新增单表、修改已有表结构 | D0/D1 可合并确认 | D0 必读 + D2 + D3 + D4 |
| 简化流程 | 加字段、加索引、修改字段属性 | D1 可省略，D4 精简 | D0 + D2 + D3 |
| 紧急流程 | 线上紧急加字段/修索引 | D1 跳过，D2 精简 | D2 编写规则不降 + D3 至少做反模式扫描 |

**降级原则**：
- DDL 编写规则（D2.2 强制项/禁止项）在任何降级级别下都不可降低
- 验证审查（D3）在任何降级级别下都不可跳过
- 降级后必须在交付输出中标注"本次执行了 X 级降级流程"及降级原因
- 紧急流程完成后，后续必须补做完整的 D0-D3 流程

```
 │                         ↓                                   │
 │                    D4: 交付输出                              │
 └─────────────────────────────────────────────────────────────┘
```

## D0: 需求分析（强制第一阶段，禁止跳过）

### D0.1 现有数据库扫描

如果是已有项目，**必须完整读取**以下内容：

```
必读：
├── 现有表结构（DDL / Entity / migration 文件）
├── 现有命名规范（表名前缀、字段命名风格）
├── 已有的公共字段模式（审计字段、软删除、租户字段）
├── 已有的主键策略（雪花ID / UUID / 其他）
├── 已有的索引命名规范
└── 数据库类型和版本（MySQL / PostgreSQL / Oracle）
```

### D0.2 业务需求拆解

将用户的业务需求拆解为数据实体和关系：

```
需求："设计一个订单系统，支持商品下单、支付、发货"

拆解：
├── 实体识别：
│   ├── 订单主表（order）
│   ├── 订单明细表（order_item）
│   ├── 收货地址表（address）
│   └── 支付记录表（payment）
│
├── 关系识别：
│   ├── 订单 1:N 订单明细
│   ├── 订单 1:1 支付记录
│   └── 订单 N:1 收货地址
│
└── 特征识别：
    ├── 订单号：业务编号，需要唯一索引
    ├── 金额：精确计算，DECIMAL
    ├── 状态：有限枚举，需状态机
    └── 时间线：多个时间节点
```

### D0.3 确认清单（使用 AskUserQuestion 上下选择）

在动手设计前，**必须向用户确认**以下内容：

使用 AskUserQuestion 工具进行多选项确认：

```markdown
【数据库设计确认】
请逐项确认以下设计规范（可上下选择）：

1. 数据库类型和版本？
   - MySQL 8.0
   - PostgreSQL 14+
   - Oracle 19c
   - 其他（请手动输入）

2. 表名前缀规范？
   - 无前缀
   - 项目缩写_（如 sys_user）
   - 模块前缀_（如 order_xxx）
   - 其他（请手动输入）

3. 公共字段模式？
   - 标准审计字段（created_by, updated_by, created_at, updated_at, deleted）
   - 简化模式（仅 created_at, updated_at, deleted）
   - 含租户字段（+ tenant_id）
   - 无公共字段
   - 自定义（请手动输入）

4. 主键策略？
   - 雪花ID（BIGINT，推荐）
   - UUID（CHAR/VARCHAR）
   - 自增ID（仅限 MySQL）
   - 业务主键

5. 预估数据量级？
   - < 10万行（无需特殊优化）
   - 10万-100万行（需合理索引）
   - 100万-1000万行（需分区规划）
   - > 1000万行（需分表/分区策略）

6. 字符集要求？
   - utf8mb4（推荐，支持 emoji）
   - utf8（仅支持基本字符）
   - 其他（请手动输入）
```

每个问题使用单独的选择题确认。

### 质量门禁 D0

- [ ] 已读取现有表结构和命名规范（已有项目）
- [ ] 已完成业务实体和关系拆解
- [ ] 已向用户确认数据库类型、命名规范、公共字段
- [ ] 能用一句话准确描述当前设计目标

## D1: 方案设计

### D1.1 实体设计

对每个实体，确定以下内容：

```
表名：{prefix}_{module}
├── 字段清单：
│   ├── 主键：id BIGINT（雪花ID）
│   ├── 业务字段：（逐个列出，标注类型和约束）
│   ├── 公共字段：（审计、软删除等）
│   └── 备注：每个字段的设计理由
│
├── 索引设计：
│   ├── 主键索引：id
│   ├── 唯一索引：业务唯一字段
│   └── 普通索引：高频查询字段
│
└── 约束设计：
    ├── NOT NULL 约束
    ├── DEFAULT 值
    ├── CHECK 约束（如需要）
    └── 外键说明（仅文档级别，物理外键视项目规范）
```

### D1.2 关系设计

```
关系类型和处理方式：

一对一（1:1）：
├── 方案A：在主表中加字段（如果从表字段少）
├── 方案B：独立从表，共享主键或唯一外键
└── 选择依据：字段数量、查询频率、是否独立变化

一对多（1:N）：
├── 方案：在"多"的一方加外键字段
├── 外键字段命名：{主表名}_id
└── 索引：外键字段必须加索引

多对多（M:N）：
├── 方案：创建关联表
├── 关联表名：{表A}_{表B}_rel 或 {描述性名称}
├── 字段：id + 表A_id + 表B_id + 扩展字段
└── 唯一约束：(表A_id, 表B_id) 联合唯一

树形结构（自引用）：
├── 方案：parent_id 自引用
├── 索引：parent_id 必须加索引
└── 深度：明确最大层级
```

### D1.3 设计评审检查

```
评审维度：
├── 完整性：是否覆盖了所有业务字段？
├── 合理性：字段类型是否合适？长度是否足够？
├── 扩展性：预留合理的扩展空间（但不过度设计）
├── 性能：索引是否覆盖高频查询？是否有冗余索引？
├── 安全：敏感数据是否需要加密存储？
└── 规范：命名、类型、约束是否符合项目规范？
```

### 质量门禁 D1

- [ ] 每个表有完整的字段清单（含类型、约束、默认值）
- [ ] 表间关系已明确（1:1 / 1:N / M:N）
- [ ] 索引已设计且有明确理由
- [ ] 符合项目已有的命名和设计规范

### D1 用户确认点

使用 AskUserQuestion 工具进行确认：

```
【表结构设计】
表1：{prefix}_{module}
├── 字段列表（类型、约束、说明）
├── 索引列表（字段、类型、理由）
└── 关系说明

表2：...

【关键设计决策】
- 决策1：选择了X方案，原因是Y
- 决策2：...
```

**使用 AskUserQuestion 工具**：
- 问题：请确认表结构设计方案
- 选项：
  1. 确认无误，开始编写 DDL
  2. 需要调整（请在下一题详细说明）

## D2: DDL 编写

### D2.1 DDL 编写顺序

```
1. 基础表（被依赖的表先建）
2. 关联表
3. 索引（建表后添加）
4. 初始数据（如需要）
5. 迁移脚本（如使用 Flyway/Liquibase）
```

### D2.2 DDL 编写规则

**强制项：**
- 每个字段必须有 COMMENT
- 每个表必须有 COMMENT
- 使用 `utf8mb4` 字符集（MySQL）
- 主键使用雪花ID（BIGINT，禁止 AUTO_INCREMENT）
- 金额使用 `DECIMAL`，禁止 `FLOAT`/`DOUBLE`
- 时间使用 `DATETIME` 或 `TIMESTAMP`（明确选择其一）
- 字段默认 `NOT NULL`，可空字段需有明确理由
- 状态/枚举字段使用 `TINYINT`/`SMALLINT` + COMMENT 列举值含义
- 字符串类型明确指定长度，禁止无限制的 `TEXT`（除非确实需要）
- 外键字段必须加索引

**禁止项：**
- 禁止使用数据库物理外键约束（`FOREIGN KEY`），通过应用层保证一致性
- 禁止使用 `ENUM` 类型（MySQL），用 `TINYINT` + COMMENT 代替
- 禁止使用保留字作为字段名
- 禁止无 COMMENT 的字段或表
- 禁止使用 `FLOAT`/`DOUBLE` 存储金额
- 禁止使用自增主键（`AUTO_INCREMENT`）
- 禁止在表中存储大对象（图片、文件内容），存储路径或 URL
- 禁止使用触发器、存储过程（业务逻辑放应用层）

### D2.3 标准 DDL 模板

```sql
-- ============================================================
-- 表：{prefix}_{module}
-- 描述：{表的业务描述}
-- 作者：{作者}
-- 日期：{创建日期}
-- ============================================================

CREATE TABLE `{prefix}_{module}` (
    -- ========== 主键 ==========
    `id`            BIGINT          NOT NULL                        COMMENT '主键（雪花ID）',

    -- ========== 业务字段 ==========
    -- （按逻辑分组排列，每组之间空一行）

    -- ========== 公共字段 ==========
    `created_by`    BIGINT          NOT NULL    DEFAULT 0           COMMENT '创建人ID',
    `updated_by`    BIGINT          NOT NULL    DEFAULT 0           COMMENT '更新人ID',
    `created_at`    DATETIME        NOT NULL    DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    `updated_at`    DATETIME        NOT NULL    DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    `deleted`       TINYINT         NOT NULL    DEFAULT 0           COMMENT '逻辑删除：0-正常，1-已删除',

    -- ========== 主键 & 索引 ==========
    PRIMARY KEY (`id`),

    -- 唯一索引
    UNIQUE KEY `uk_{表名简写}_{字段名}` (`唯一字段`),

    -- 普通索引
    KEY `idx_{表名简写}_{字段名}` (`索引字段`),

    -- 联合索引（遵循最左前缀原则）
    KEY `idx_{表名简写}_{字段1}_{字段2}` (`字段1`, `字段2`)

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci
  COMMENT='{表的业务描述}';
```

### 质量门禁 D2

- [ ] 所有字段有 COMMENT
- [ ] 表有 COMMENT
- [ ] 字段类型和长度合理
- [ ] 索引命名符合规范
- [ ] 无禁止项
- [ ] DDL 可直接执行

## D3: 验证审查

### D3.1 自动检查清单

```
逐项检查：
├── 命名规范
│   ├── 表名：{prefix}_{module}，snake_case
│   ├── 字段名：snake_case
│   ├── 索引名：uk_/idx_ 前缀 + 表名简写 + 字段名
│   └── 无保留字冲突
│
├── 字段规范
│   ├── 类型是否合理（金额DECIMAL、状态TINYINT、时间DATETIME）
│   ├── 长度是否足够（VARCHAR预留20%-30%空间）
│   ├── NOT NULL 默认设置是否合理
│   ├── DEFAULT 值是否合理
│   └── COMMENT 是否完整
│
├── 索引规范
│   ├── 主键索引 ✓
│   ├── 唯一索引：业务唯一字段有唯一索引
│   ├── 外键字段有索引
│   ├── 联合索引遵循最左前缀
│   └── 无冗余索引
│
├── 关系规范
│   ├── 外键字段命名一致（{表名}_id）
│   ├── 外键字段有索引
│   └── 无物理外键约束
│
└── 性能预判
    ├── 预估数据量下的查询效率
    ├── 是否需要分区表（>1000万行考虑）
    └── 是否有全表扫描风险
```

### D3.2 反模式扫描

对照 `references/anti-patterns.md` 逐项扫描，确保无反模式。

### 质量门禁 D3

- [ ] 自动检查清单全部通过
- [ ] 无反模式
- [ ] 与用户需求完全对齐

## D4: 交付输出

### D4.1 SQL 文件输出路径规范（强制）

所有数据库 SQL 文件必须按以下规则存放：

```
根目录/
└── sql/
    ├── ddl/          -- 数据定义语言（建表、修改表结构、索引）
    │   ├── V{version}__{描述}.sql          -- Flyway/Liquibase 迁移脚本
    │   └── {table_name}__create.sql         -- 单表建表脚本（可选）
    │
    └── dml/          -- 数据操作语言（增删改数据）
        ├── {table_name}__insert_initial.sql -- 初始数据
        ├── {table_name}__update_{字段}.sql  -- 数据更新脚本
        └── V{version}__{描述}.sql           -- 数据迁移脚本
```

**命名规范：**
- 迁移脚本：`V{版本号}__{简短描述}.sql`（双下划线分隔）
- 单表脚本：`{表名}__{操作类型}.sql`
- 版本号格式：`001`、`002` 或 `YYYYMMDDNNN`

### D4.2 交付物清单

```
1. DDL 脚本（存放于 sql/ddl/）
   ├── 建表语句（带完整 COMMENT 和索引）
   ├── 初始数据 INSERT（如需要）
   └── 迁移脚本命名（Flyway: V{version}__{description}.sql）

2. 设计文档
   ├── ER 关系说明
   ├── 表结构说明
   ├── 索引设计说明
   └── 设计决策记录

3. 变更影响分析（修改已有表时）
   ├── 新增/修改的字段
   ├── 新增/修改的索引
   ├── 对现有数据的影响
   └── 数据迁移方案（如需要）
```

### D4.3 交付确认

向用户展示最终结果，包含：

```
【变更摘要】
新建/修改了哪些表，每张表的用途

【表结构】
完整的字段列表 + 类型 + 约束 + 说明

【索引设计】
每个索引的字段、类型、设计理由

【设计决策】
关键设计选择及原因

【后续建议】
后续可优化但不在本次范围的点
```

## 协作与辅助 Skill

### 上游 Skill：tk-dev-task-precheck

本 skill 支持被 `tk-dev-task-precheck`（开发任务前置守门员）作为下游 skill 调用。
当通过 `tk-dev-task-precheck` 交接触发时：
- 交接包中的项目技术栈快照、数据库类型信息可作为 D0 阶段的参考输入
- 但 D0 的现有数据库扫描和需求确认仍然是强制的，不可跳过
- D0 的用户确认点仍然需要执行，不可跳过
- 如果交接包中包含了前置分析结果，D0.1（现有数据库扫描）可快速确认而非从头扫描

### 下游 Skill 协作

数据库设计完成后，结果可交接给下游 skill：
- `tk-springboot-dev-standards` — 基于表结构生成 Entity/Mapper/Service
- `tk-vue3-dev-standards` — 基于表结构设计前端表单和列表页

协作时，本 skill 的 D4 交付物（DDL 脚本 + 设计文档）应作为下游 skill 的输入。
