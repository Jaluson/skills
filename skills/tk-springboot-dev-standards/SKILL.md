# Spring Boot 标准化开发规范 — AI 自动化执行流水线

## 前置条件：环境检测（强制）

在执行 Spring Boot 开发任务之前，**必须先完成环境检测**：

```
┌─────────────────────────────────────────────┐
│  1. 调用 tk-dev-env-manager 执行环境检测      │
│     - 检测 JDK 是否安装及版本                 │
│     - 检测 Maven/Gradle 是否安装              │
│  2. 环境检测通过 → 继续执行 P0 阶段           │
│  3. 环境检测未通过 → 完成环境配置后再继续      │
└─────────────────────────────────────────────┘
```

**未通过环境检测的代码不得交付。**

## 数据库协作（强制）

当任务涉及以下数据库相关工作时，**必须先协作 tk-database-design-standards**：

- 设计新的数据库表结构
- 修改已有表的结构（添加/修改/删除字段）
- 创建新的 Entity 实体类
- 编写数据库迁移脚本
- 设计数据库索引

**协作流程**：
```
1. 识别到数据库相关任务
2. 调用 tk-database-design-standards 执行数据库设计
3. 获取 DDL 输出和设计文档
4. 基于 DDL 生成 Entity/Mapper/Service/Controller
5. 继续执行 P3 及后续阶段
```

本 skill 是 AI 执行 Spring Boot 开发任务时**必须严格遵循的执行流水线**。
每个阶段都有明确的输入、输出和质量门禁，只有通过当前门禁才能进入下一阶段。

### 三条不可违反的铁律

**铁律一：不明确就提问，禁止猜测。**
需求中任何模糊、有歧义、或可以有多种理解的表述，都必须向用户提问澄清。
宁可多问一句，也不能凭猜测写出一堆不符合用户预期的代码。
判断标准：如果对一个需求点你能想到两种及以上的理解方式，就必须提问。

**铁律二：分析完毕必须展示理解，等待用户确认后才动手。**
每完成一个分析阶段（P0/P1/P2），必须向用户主动展示你对需求的理解、
你的发现和你的计划。用户说"可以""没问题""继续"后才能进入下一阶段。
禁止跳过确认直接开始编码。

**当用户要求跳过确认时**：用户说"别问了直接做""快速搞一下""跳过吧"等，
按以下规则处理：

| 确认点 | 用户说跳过时 | 原因 |
|--------|------------|------|
| P0 需求澄清 | **可合并**：将 P0/P1/P2 合并为一次展示 | 需求已足够清晰时，三次确认确实冗余 |
| P1 变更范围 | **可合并**：与 P2 合并展示 | 信息重叠度高 |
| P2 方案设计 | **不可跳过**：必须展示方案并等待确认 | 这是编码前的最后防线，必须确保方向正确 |

合并后的最小确认格式：
```
【需求 + 方案确认（合并展示）】
需求理解：（一句话）
方案概述：（两三句话）
关键设计决策：（列出非直觉的选择）
变更文件：（列出）

确认后开始编码。
```
即使合并，P2 的方案确认点也**不可省略**。

**铁律三：动手前检查，有没有相关 skill 可以辅助。**
在进入编码之前，检查当前环境是否有可用的相关 skill（如代码审查、测试、
部署等）。如果有，在合适的阶段主动使用它们，而不是全部手动处理。
特别注意：如果存在 `tk-dev-task-precheck` skill，编码任务应优先通过它作为入口，
由它完成前置分析后再交接给本 skill，而非直接触发本 skill。
如果本 skill 是被 `tk-dev-task-precheck` 交接触发的，接收交接包中的预扫描结果
作为参考输入，但本 skill 的 P0 深度阅读阶段仍然是强制的，不可跳过。

**铁律四：变更后必须验证。** 任何文件变更（新建/修改/删除代码文件、配置文件等）完成后，
必须执行编译校验验证变更是否正确，并重新确认是否满足用户的原始需求。
编译失败或需求不满足的代码不得交付。

**铁律五：提问后必须重新确认。** 向用户提问获取回答后，必须基于回答重新出具完整方案，并将该方案以菜单选项形式交给用户最终确认。用户选择确认选项后，方可进入下一步。未获得用户确认的方案不得执行。

**铁律六：方案必须包含详细的改动信息。** 提交给用户确认的方案中，对于每个文件变更必须包含：
- **新建 Java 类**：完整包路径+类名、所属层级（Controller/Service/Mapper/Entity/DTO/VO等）、核心职责、一句话说明关键逻辑
- **新建配置文件**：文件完整路径、配置项名称及用途
- **修改文件**：文件完整路径、修改内容摘要、为何需要修改
- **删除文件**：文件完整路径、删除原因

示例格式：
```
【文件变更清单】
新增 Java 类：
  - com.example.user.controller.UserController          — 用户管理 Controller，处理用户 CRUD 接口
  - com.example.user.service.UserService                — 用户业务逻辑，封装用户相关操作
  - com.example.user.service.impl.UserServiceImpl       — 用户业务实现，包含具体业务逻辑
  - com.example.user.mapper.UserMapper                  — 用户数据访问层
  - com.example.user.entity.User                       — 用户实体类，与数据库表对应
  - com.example.user.dto.CreateUserRequest              — 创建用户请求DTO
  - com.example.user.dto.UserResponse                  — 用户响应DTO

新增配置文件：
  - src/main/resources/mapper/UserMapper.xml            — MyBatis SQL 映射文件

修改文件：
  - src/main/java/com/example/common/Result.java        — 新增 errorCode 字段，完善错误码体系

删除文件：
  - src/main/java/com/example/user/util/UserUtil.java   — 已废弃，合并到 UserService 中
```

禁止仅用"新建用户管理模块"等模糊描述，必须逐项列出具体的类名、包路径、层级和核心职责。

**允许并行 Agent 加速。** 在 P0-P3 阶段中，当任务涉及多个文件或多项检测时，允许启用最多 5 个子 agent 并行执行以加速任务：
- P0 阶段：多个文件的深度阅读可以并行
- P1 阶段：多个模块的模式分析可以并行
- P2 阶段：多个接口的方案设计可以并行
- P3 阶段：多个文件的编码可以并行

子 agent 必须有明确的输入、输出和职责边界，只负责只读操作（扫描/检测）或指定任务的执行，不负责写操作（编码/修改文件）之外的决策。

### 流程降级机制

标准流程是完整 7 阶段，但允许根据**任务复杂度**降级，降级不等于放弃质量：

| 降级级别 | 适用场景 | 可精简的阶段 | 不可降级的底线 |
|----------|---------|-------------|---------------|
| 完整流程 | 新增模块、重构、涉及共享接口变更 | 无 | 全部 P0-P6 |
| 标准流程 | 新增 API/Service、修改涉及 ≤3 文件 | P0/P1 可合并确认 | P0 必读 + P3 + P4 + P5 + P6 |
| 简化流程 | 改动 ≤3 行、无新文件、无依赖变更 | P1/P2 可省略，P6 精简 | P0 + P3 + P4 |
| 紧急流程 | 线上 hotfix | P1 可跳过，P2 精简 | P3 编码标准不降 + P4 至少做编译检查 |

**降级原则**：
- 编码规则（P3.2 禁止项/强制项）在任何降级级别下都不可降低
- 编译验证（P4）在任何降级级别下都不可跳过
- 降级后必须在交付输出中标注"本次执行了 X 级降级流程"及降级原因
- 紧急流程完成后，后续必须补做完整的 P0-P5 流程

## P0: 深度阅读（强制第一阶段，禁止跳过）

### P0.1 项目结构扫描

扫描项目根目录和源码目录，获取全貌：
- 构建工具（Maven `pom.xml` / Gradle `build.gradle`）
- Spring Boot 版本和关键依赖版本
- Java 版本（`sourceCompatibility` 或 `java.version`）
- 配置文件（`application.yml` / `application.properties`，含 profile 配置）
- 项目模块结构（单模块 or 多模块）
- **依赖版本管理方式**：确认是否使用统一版本管理（Maven `dependencyManagement` / Gradle `version catalog` 或 `buildSrc`）

### P0.2 技术栈识别

锁定项目使用的技术栈（不是推荐，是项目已有的事实）：

| 维度 | 检测方式 | 关注点 |
|------|----------|--------|
| 数据访问层 | pom.xml 依赖 | MyBatis / MyBatis-Plus / JPA / JDBC |
| 数据库 | application.yml | MySQL / PostgreSQL / Oracle / H2 |
| 缓存 | pom.xml + 配置 | Redis / Caffeine / 无 |
| 消息队列 | pom.xml + 配置 | RabbitMQ / Kafka / 无 |
| 安全框架 | pom.xml | Spring Security / Sa-Token / 自定义 |
| API 文档 | pom.xml | Swagger / SpringDoc / 无 |
| 工具库 | pom.xml | Lombok / MapStruct / Hutool / Guava |

### P0.3 目标区域深度阅读

根据任务类型，**必须完整阅读**以下文件：

**新增功能时读取：**
- 目标模块的目录下所有相关文件
- 已有的 Entity/DTO/VO 类（理解数据模型）
- 已有的 Mapper/Repository（理解数据访问模式）
- 已有的 Service 层（理解业务逻辑模式）
- 已有的 Controller（理解 API 规范）
- 配置文件中的相关配置

**修改已有功能时读取：**
- 要修改的文件**完整内容**
- 该文件 import 的所有直接依赖
- 该文件被哪些其他文件引用
- 相关的测试文件（如果存在）

**修改数据库相关时额外读取：**
- 现有的 migration 文件（Flyway/Liquibase）
- 或现有的 SQL 初始化脚本
- 相关的 Entity 类和表结构

### P0.4 已有复用资源盘点

- 已有的公共工具类（`utils/` 或 `common/`）
- 已有的公共常量类
- 已有的公共异常类和全局异常处理器
- 已有的基础 Entity（如 `BaseEntity` 含公共字段）
- 已有的公共配置（如 MyBatis 配置、Redis 配置）
- 已有的拦截器/过滤器

### 质量门禁 P0

- [ ] 已读取构建配置确认技术栈和版本
- [ ] 已确认依赖是否使用统一版本管理（dependencyManagement / version catalog）
- [ ] 已读取 application.yml/properties 确认项目配置
- [ ] 已完整阅读所有直接相关的源文件
- [ ] 已确认被修改文件的所有调用方
- [ ] 已盘点可复用的工具类、基础类、公共配置
- [ ] 能用一句话准确描述当前需求的本质

### P0 用户确认点：需求澄清

P0 完成后，**必须向用户展示以下内容并等待确认**：

```
【需求理解】
我理解您需要的是：（用用户能听懂的自然语言描述需求本质）

【已发现的不明确点】
在阅读项目代码后，以下几点我需要向您确认：
1. （具体的不明确点 + 你的理解 + 请用户选择或补充）
2. ...

【已识别的现有资源】
- 可复用工具类/基类：（列出）
- 相关已有代码：（列出）

```

## P1: 全局分析

### P1.1 项目模式锁定

确定项目遵循的模式：
- **分层结构**：controller → service → mapper/dao → entity
- **包组织方式**：按层分包（controller/service/dao/entity）或按模块分包（user/[controller,service,dao]）
- **命名风格**：已有类的命名模式
- **返回值封装**：统一的 `Result<T>` / `ApiResponse<T>` / 自定义响应类
- **分页模式**：PageHelper / MyBatis-Plus Page / Spring Data Page
- **异常处理**：全局异常处理器 + 自定义异常类
- **参数校验**：JSR 303 Validation / 手动校验

### P1.2 变更范围分析

```
必须新建的文件（列表）:
  └── 文件路径 + 职责说明

必须修改的文件（列表）:
  └── 文件路径 + 修改内容 + 确认不影响其他调用方

可能影响的文件（需验证）:
  └── 文件路径 + 需要验证的点

明确不需要修改的文件:
  └── 可能被误改但实际不需要动的文件
```

### 质量门禁 P1

- [ ] 项目模式已锁定（分层、包组织、返回值封装、异常处理）
- [ ] 变更范围清单已完成
- [ ] 风险点已识别
- [ ] 确认变更不会破坏已有功能

### P1 用户确认点：变更范围确认

P1 完成后，**必须向用户展示以下内容并等待确认**：

```
【项目模式确认】
分层方式、包组织、返回值封装、异常处理、分页方式等

【变更范围】
需要新建的文件：（列出路径 + 用途）
需要修改的文件：（列出路径 + 修改内容）
不需要改动的文件：（列出排除项）

【风险评估】
潜在风险点：（列出）→ 保护措施：（说明）
```

**使用 AskUserQuestion 工具进行确认**：

问题：变更范围是否正确？

```
- 确认无误，继续进行方案设计
- 需要调整（请在下一题详细说明）
```

- 用户确认后才能进入 P2
- 如果用户有调整 → 回到 P0/P1 补充分析

## P2: 方案设计

### P2.1 分层设计

对每个要新建或修改的文件，确定其在分层架构中的位置：

```
Controller 层：接收请求、参数校验、调用 Service、返回响应
    ↓
Service 层：业务逻辑、事务管理、调用 Mapper/外部服务
    ↓
Mapper/DAO 层：数据访问、SQL/CRUD
    ↓
Entity 层：数据模型，与数据库表对应

辅助层：
├── DTO：数据传输对象（跨层传输、接口参数）
├── VO：视图对象（返回给前端的数据结构）
├── Converter：DTO ↔ Entity ↔ VO 的转换
├── Config：配置类
└── Utils/Constants：工具和常量
```

### P2.2 接口设计

确定 API 接口的完整定义：
- URL 路径和 HTTP 方法
- 请求参数（路径参数、查询参数、请求体）
- 响应格式（统一封装 + 具体数据结构）
- 状态码（成功、各种错误场景）

### P2.3 数据设计

确定数据相关的定义：
- Entity 类（字段与数据库列的映射）
- DTO 类（接口入参的结构）
- VO 类（接口出参的结构）
- Mapper/Repository 接口（需要的查询方法）
- SQL（复杂查询需要预先设计 SQL）

### 质量门禁 P2

- [ ] 每个新建/修改文件有明确的职责和所属层
- [ ] API 接口已定义（URL、方法、参数、响应）
- [ ] 数据模型已设计（Entity、DTO、VO）
- [ ] 与项目已有模式一致
- [ ] 设计决策有理由

### P2 用户确认点：方案设计确认（进入编码前的最后一道确认）

P2 完成后，**必须向用户展示以下内容并等待确认**：

```
【方案概述】
（简明扼要描述整体实现方案）

【文件清单与职责】
文件路径 — 层级 — 职责描述
...

【API 接口设计】
- POST   /api/xxx     — 创建xxx
- GET    /api/xxx     — 查询列表
...

【数据模型】
Entity: 字段列表
DTO: 入参字段
VO: 出参字段

【关键设计决策】
- 决策 1：选择了 X 方案，原因是 Y
- 决策 2：...

【待确认问题】
（设计中需要用户拍板的选择）
```

**使用 AskUserQuestion 工具进行确认**：

问题：方案设计确认

```
- 确认无误，开始编码
- 需要调整（请在下一题详细说明）
```

- 用户确认后才能进入 P3 编码
- 这是编码前的最后一道防线，确认后才开始动手
- 如果用户有调整 → 更新 P2 设计并重新确认

## P3: 编码实现

### P3.1 编码顺序（严格按依赖方向）

```
1. Entity → 定义数据模型，所有后续代码依赖于此
2. DTO/VO → 定义接口数据结构
3. Mapper/DAO → 定义数据访问接口
4. Mapper XML（如使用 MyBatis）→ 编写 SQL
5. Service → 实现业务逻辑
6. Controller → 暴露 API 接口
7. 配置/注册 → 将新模块接入系统（如需要）
```

### P3.2 编码规则

**禁止项：**
- 禁止在 Controller 中写业务逻辑
- 禁止在 Service 中直接拼接 SQL（应在 Mapper/DAO 层）
- 禁止吞掉异常（空 catch 块或只打印日志不处理）
- 禁止硬编码配置值（用 `@Value` 或配置类）
- 禁止魔法值（用常量或枚举）
- 禁止 `System.out.println`（用日志框架）
- 禁止直接返回 Entity 给前端（用 VO）
- 禁止忽略参数校验
- 禁止使用数据库自增主键（`IdType.AUTO` / `GenerationType.IDENTITY`），必须使用雪花ID（`IdType.ASSIGN_ID`）
- 禁止用 `SELECT COUNT(*)` 或查完整记录判断是否存在（使用 `EXISTS` 或 `SELECT 1 ... LIMIT 1`）
- 禁止新增/修改/删除操作不加事务保护
- 禁止在 Service 中写格式校验代码（非空、长度、格式等应通过 DTO 注解完成）

**强制项：**
- Controller 方法必须用 `@Valid` / `@Validated` 触发 DTO 注解校验
- 参数校验优先使用注解（`@NotBlank`, `@Size`, `@Email`, `@Pattern`, 自定义注解等）
- 创建和更新校验规则不同时，必须使用校验分组（`@Validated(Create.class)`）
- 能用自定义注解封装的校验（如唯一性校验 `@UniqueUsername`），不在 Service 中手写 if-else
- 新增/修改/删除操作必须加 `@Transactional(rollbackFor = Exception.class)` 确保原子性
- 存在性检查必须使用 `EXISTS` 而非 `SELECT COUNT(*)`
- 公共方法必须有 Javadoc
- 异常通过全局异常处理器统一处理
- 使用项目已有的统一返回值封装
- 日志使用 SLF4J，不直接使用 Log4j/Logback
- **依赖版本必须使用统一版本管理**：Maven 项目通过 `dependencyManagement` 声明版本，子模块引用时不带版本号；Gradle 项目使用 `version catalog`（`libs.versions.toml`）或 `buildSrc` 集中管理版本

### P3.3 文件级微自检

每写完一个文件，立即检查：
1. 导入是否正确，无缺失无多余
2. 注解是否完整（`@Service`、`@RestController`、`@Mapper` 等）
3. 依赖注入方式是否与项目一致（`@Autowired` / 构造器注入）
4. 方法签名与 P2 设计一致
5. 命名是否符合规范

### 质量门禁 P3

- [ ] 所有文件按依赖顺序编写
- [ ] 每个文件通过了微自检
- [ ] 无禁止项
- [ ] 所有强制项已满足
- [ ] 代码与 P2 设计方案一致

## P4: 编译验证与自动修复（强制）

### P4.1 执行验证命令

```bash
# 1. 编译检查
mvn compile          # Maven
# 或
gradle compileJava   # Gradle

# 2. 测试（如果有相关测试）
mvn test
# 或
gradle test

# 3. 完整构建
mvn package -DskipTests=false
# 或
gradle build
```

### P4.2 错误分类与修复

**致命错误（必须修复）：**
- 编译错误（语法、类型不匹配、找不到符号）→ 修正代码
- 测试失败 → 分析失败原因并修复

**重要警告（应该修复）：**
- 未使用的导入 → 清理
- 依赖警告 → 检查版本兼容性

### P4.3 自动修复循环

```
最多 5 轮：
  1. 执行编译/测试
  2. 收集错误
  3. 全部通过 → 退出
  4. 按文件分组修复
  5. 验证修复
  6. 回到步骤 1

5 轮后仍有错误 → 报告给用户
```

### P4.4 常见错误快速修复

| 错误类型 | 常见原因 | 修复方法 |
|----------|----------|----------|
| 找不到符号 | 导入缺失或路径错误 | 添加 import 或修正类名 |
| 类型不匹配 | 泛型或类型转换问题 | 修正类型声明 |
| 缺少注解 | 未标注 `@Service`/`@Mapper` 等 | 添加相应注解 |
| Bean 注入失败 | 组件未注册或名称不匹配 | 添加注解或修正 Bean 名称 |
| SQL 错误 | MyBatis XML 中 SQL 语法或字段不匹配 | 修正 SQL |
| 依赖冲突 | 版本不兼容 | 排查依赖树 |

### 质量门禁 P4

- [ ] 编译零错误
- [ ] 测试全部通过（如存在相关测试）
- [ ] 构建成功
- [ ] 所有修复都是根因修复

## P5: 需求回验（强制，编译通过后必须执行）

**目的**：代码编译通过不代表需求已完成。此阶段要求回到原始需求本身，逐条验证每个需求点
是否被正确实现。不满足则自动回退修复，形成闭环。

这是整个流水线中最关键的"守门员"——前四个阶段保证代码"写得对"，P5 保证代码"做对了事"。

### P5.1 需求拆解与清单化

重新阅读用户的原始需求，将其拆解为可验证的原子化需求点：

```
原始需求："写一个订单管理的增删改查接口，支持按状态筛选、分页查询、导出Excel"

拆解为需求清单：
├── R1: POST   /api/orders          — 创建订单
├── R2: DELETE /api/orders/{id}     — 删除订单
├── R3: PUT    /api/orders/{id}     — 更新订单
├── R4: GET    /api/orders          — 分页查询订单列表
├── R5: GET    /api/orders/{id}     — 查询订单详情
├── R6: GET    /api/orders          — 按状态筛选（查询参数）
└── R7: GET    /api/orders/export   — 导出 Excel
```

### P5.2 逐条需求验证

对每个需求点，执行以下验证：

```
对于每个需求点 Rn：
  1. 在代码中定位对应的实现位置（Controller → Service → Mapper）
  2. 追踪该需求的完整调用链
  3. 验证以下维度：

  功能维度：
  ├── 接口路径和 HTTP 方法是否正确？
  ├── 请求参数是否完整？校验注解是否到位？
  ├── 返回值是否正确封装？数据结构是否完整？
  ├── 异常场景是否处理？（参数错误、数据不存在、约束冲突）
  └── 日志是否记录关键操作？

  数据维度：
  ├── Entity 字段映射是否完整？
  ├── DTO → Entity 转换是否有遗漏字段？
  ├── Entity → VO 转换是否正确？
  └── SQL 是否正确（查询条件、排序、分页）？
```

### P5.3 差距分析与自动修复

```
需求回验循环（最多 3 轮）：
  ┌──────────────────────────────────────────────┐
  │  1. 逐条验证所有需求点                         │
  │  2. 标记每个需求点的状态：                       │
  │     ✅ 满足 — 功能完整，无需修改                 │
  │     ⚠️ 部分满足 — 有实现但存在遗漏               │
  │     ❌ 缺失 — 完全未实现                         │
  │  3. 如果全部 ✅ → 通过，进入 P6                  │
  │  4. 收集所有 ⚠️ 和 ❌ 的需求点                   │
  │  5. 按优先级排序：❌ 优先于 ⚠️                   │
  │  6. 回退到 P3，补充实现缺失的功能                │
  │  7. 重新执行 P4 编译验证                        │
  │  8. 回到步骤 1，重新验证                         │
  │                                               │
  │  如果 3 轮后仍有未满足的需求：                    │
  │  → 暂停，向用户报告差距清单，等待用户指示         │
  └──────────────────────────────────────────────┘
```

**用户通知规则**：
- 每次回退循环开始前，**必须**向用户简要说明发现的问题和即将进行的修复
- 修复不涉及设计变更时（如遗漏了一个参数校验），通知后直接修复，不需要等确认
- 修复涉及设计变更时（如发现需要新增一个 Service 或改变数据结构），必须回退到 P2 重新确认
- 修复完成后，向用户报告修复结果

### P5.4 常见需求遗漏检查

回验时**逐项确认**以下易遗漏点：

```markdown
接口层面：
- [ ] 每个 API 的 HTTP 方法是否正确（GET/POST/PUT/DELETE）
- [ ] 每个 API 的请求参数是否有 @Valid 校验
- [ ] 每个 API 的返回值是否用统一封装（Result<T>）
- [ ] 每个 API 是否有对应的异常处理

数据层面：
- [ ] Entity 与数据库字段映射是否完整
- [ ] DTO 缺少必要字段？
- [ ] VO 是否遗漏前端需要的字段？
- [ ] 分页参数是否正确传递和返回？
- [ ] 排序是否生效？

业务层面：
- [ ] 唯一性校验（用户名/编码不能重复等）
- [ ] 关联数据一致性（删除主表时子表处理）
- [ ] 状态流转是否合法（如订单状态只能向前流转）
- [ ] 并发场景是否考虑（乐观锁/分布式锁）
- [ ] 操作日志是否记录

性能层面：
- [ ] 存在性检查使用 EXISTS 而非 SELECT COUNT(*)
- [ ] 查询是否避免不必要的字段和数据量
- [ ] 批量操作是否使用批量接口

事务安全层面：
- [ ] 所有新增/修改/删除接口都有事务保护
- [ ] 事务注解指定了 rollbackFor = Exception.class
- [ ] 涉及多表写操作的接口在同一事务中
- [ ] 事务边界合理，不包含不必要的远程调用
```

### 质量门禁 P5

- [ ] 原始需求已拆解为原子化需求清单
- [ ] 每个需求点已逐条验证并标记状态
- [ ] 所有需求点状态为 ✅ 满足
- [ ] 无功能性遗漏（接口、数据、业务规则）
- [ ] 回验循环未超过 3 轮

> **铁律：未通过 P5 的代码，即使编译通过也绝不可交付。编译正确 ≠ 做对了事。**

> 如果 P5 超过 3 轮仍有未满足需求：**暂停并向用户报告**，列出具体差距和建议。

### P6.1 代码质量终审

## 核心设计原则

### 最小改动原则
每次变更必须是最小必要改动，不做不相关的"顺手优化"。

### 分层架构原则
严格遵循 Controller → Service → Mapper/DAO 的单向依赖，禁止反向依赖或跨层调用。

### 统一规范原则
遵循项目已有的模式（返回值封装、异常处理、命名风格），不引入新的模式。

### 自动化驱动

- 需求分析和方案设计阶段：**必须与用户确认**，不擅自行动
- 技术选择和风格适配：自动完成，无需人工确认
- 编码后：自动验证、自动修复、自动优化
- 遇到无法自动解决的问题才上报用户

## 协作与辅助 Skill

在执行任务的过程中，应主动检查当前环境中是否有可用的辅助 skill，
并在合适的阶段使用它们，而不是全部手动处理。

### 上游 Skill：tk-dev-task-precheck

本 skill 支持被 `tk-dev-task-precheck`（开发任务前置守门员）作为下游 skill 调用。
当通过 `tk-dev-task-precheck` 交接触发时：
- 交接包中的项目技术栈快照、目录结构概览、资源盘点可作为 P0 阶段的参考输入
- 但 P0 的深度阅读（逐行理解目标区域代码）仍然是强制的，不可跳过
- P0 的用户确认点仍然需要执行，不可跳过
- 如果交接包中包含了前置分析结果，P0.1（项目结构扫描）可快速确认而非从头扫描

### Skill 使用策略

在以下时机检查并使用相关 skill：

| 时机 | 可能相关的 skill | 使用方式 |
|------|-----------------|----------|
| P0 深度阅读 | 项目探索类 skill | 辅助快速理解项目结构 |
| P2 方案设计 | 架构设计类 skill | 获取架构建议 |
| P3 编码实现 | 代码生成/模板类 skill | 提高编码效率 |
| P4 编译验证 | 测试/lint 类 skill | 自动化验证和修复 |
| P6 最终审查 | 代码审查类 skill | 获取代码质量建议 |

### 使用原则

- **优先使用已有 skill**：如果环境中存在能完成某个子任务的 skill，优先使用它
- **不重复造轮子**：skill 已经封装好的能力不需要手动实现
- **合理组合**：多个 skill 可以组合使用，但要注意上下文衔接
- **不强行使用**：如果没有合适的 skill 或 skill 不适用，按本规范手动处理
