## 任务类型识别

收到需求后，首先判断任务类型，不同类型有不同的必经步骤：

| 任务类型 | 标志 | P0 重点 | 额外注意事项 |
|----------|------|---------|-------------|
| **全新项目** | "搭个项目" "创建项目" "从零开始" | 目标目录是否为空 | 走初始化流程，不走标准流水线 |
| **Nuxt/SSR** | 项目有 `nuxt.config.*` | SSR 特有约定（见下方说明） | auto-imports、服务端数据获取、hydration |
| 新建页面 | "新增一个页面" "创建视图" | 现有页面结构、路由模式、布局模式 | 路由注册、布局选择 |
| CRUD 模块 | "管理xxx" "增删改查" | 现有 CRUD 模块（参考模式）、API 定义 | 全套文件创建 |
| 修改功能 | "修改xxx" "调整xxx" | 目标文件完整代码、调用方 | 最小改动、回归验证 |
| 通用组件 | "做一个可复用的xxx" | 现有通用组件（避免重复） | 接口设计、泛型考虑 |
| 重构 | "重构xxx" "优化xxx" | 完整依赖图、所有调用方 | 小步前进、每步验证 |
| Bug 修复 | "修复xxx" "xxx不工作" | 问题代码、复现条件 | 根因分析、回归测试 |

### Nuxt / SSR 项目的额外注意事项

当 P0 检测到项目是 Nuxt（存在 `nuxt.config.*`）时，以下规则覆盖标准规范：

```
Nuxt 特有约束：
├── auto-imports：composables/ 和 components/ 下的文件自动注册
│   ├── 禁止手动 import 自动注册的组件/composable
│   └── 新建 composable 时确认它会被自动注册
├── 文件路由：pages/ 目录即路由，禁止手动配置路由
│   ├── 新页面放在 pages/ 下，文件路径即路由路径
│   └── 动态路由用 [id].vue，嵌套用目录结构
├── 数据获取：必须用 useFetch / useAsyncData 而非自行封装
│   ├── 服务端渲染时，数据在 server 端获取并序列化到客户端
│   ├── 禁止在 onMounted 中做首次数据获取（会变成 CSR-only）
│   └── useFetch 的 key 必须唯一，避免数据冲突
├── 服务端限制：
│   ├── 组件的 setup 阶段不能访问 window / document / localStorage
│   ├── 需要浏览器 API 的代码必须用 onMounted 包裹
│   └── 第三方库如果依赖浏览器环境，需动态 import
├── Hydration 注意：
│   ├── 服务端和客户端渲染结果必须一致，否则 hydration mismatch
│   ├── 禁止在 setup 中使用 Date.now() / Math.random() 等不确定值
│   └── v-html 在 SSR 中有安全风险，优先用 v-text
└── 服务器 API：
    ├── server/api/ 目录即 API 路由
    └── 如果项目有 server 目录，API 调用可能走内部路由而非外部服务
```

## 新建页面流程

### P0：阅读清单

```
必读文件：
├── src/router/index.ts（或路由目录）     # 了解路由模式
├── 已有页面组件 1-2 个（同类型）         # 了解页面结构模式
├── src/App.vue                          # 了解布局方式
├── src/main.ts                          # 了解插件注册
├── 相关的 API/composable 文件            # 了解数据获取模式
└── styles/variables.css（如存在）        # 了解 CSS 变量体系

按需读取：
├── src/layouts/（如存在）               # 了解布局组件
├── 全局类型文件                          # 了解可用类型
└── shared/components/                   # 了解可用通用组件

UI 一致性检查（P0 阶段必须完成）：
├── 找到 1 个最相似的已有页面作为参考模板
├── 记录其使用的布局结构、组件、交互模式
├── 记录其使用的 CSS 变量（颜色、间距、字号）
└── 盘点可直接复用的通用组件（表格、表单、搜索栏等）
```

### P2：设计清单

```
设计项：
├── 页面路由（路径、名称、meta、参数）
├── 页面布局（使用哪个 Layout）
├── 数据获取（用哪个 composable/API）
├── 子组件划分（页面内的组件拆分）
├── 状态管理（页面级状态放哪）
├── 路由守卫（是否需要权限控制）
└── UI 一致性确认：
    ├── 布局结构是否与参考页面一致？
    ├── 交互模式（弹窗 vs 抽屉）是否与项目统一？
    ├── 是否复用了已有通用组件？
    └── 视觉样式是否使用 CSS 变量？
```

### P3：编码顺序

```
1. types.ts           → 页面相关类型定义（如有新类型）
2. api/*.ts           → API 函数（如有新接口）
3. composables/*.ts   → 数据获取 composable（如需要）
4. 子组件              → 页面内的展示组件
5. 页面组件            → 组装子组件
6. router/index.ts    → 注册路由（最后一步）
```

### P4：验证重点

- 路由是否正确注册，能否正常导航
- 页面是否正确使用了布局
- 数据获取是否正常（检查 loading/error 状态）
- 构建是否成功
- UI 一致性验证：
  - 样式中无硬编码颜色/间距（grep 检查）
  - 布局结构与参考页面一致
  - 交互模式与项目统一

## 新增 CRUD 模块流程

### P0：阅读清单

```
必读文件：
├── 现有 CRUD 模块的完整代码（选一个最典型的作为参考）
│   ├── types.ts
│   ├── api/*.ts
│   ├── composables/*.ts
│   ├── components/*.vue（列表、表单、详情）
│   └── index.ts
├── src/router/                        # 了解路由模式
├── src/shared/components/             # 盘点可复用组件
└── src/types/                         # 了解基础类型

重点观察：
├── 列表页的筛选/分页/排序模式
├── 表单的验证/提交模式
├── 删除确认的交互模式
└── API 函数的命名和参数模式
```

### P2：设计清单

```
文件清单（标准 CRUD 模块）：
features/{module}/
├── types.ts              → 类型定义
│   ├── {Entity}Info      → 实体类型
│   ├── {Entity}ListParams → 列表查询参数
│   ├── Create{Entity}Params → 创建参数
│   └── Update{Entity}Params → 更新参数
├── api/
│   └── {entity}.ts       → CRUD API 函数
│       ├── get{Entity}List()   → 列表查询
│       ├── get{Entity}()       → 详情查询
│       ├── create{Entity}()    → 创建
│       ├── update{Entity}()    → 更新
│       └── delete{Entity}()    → 删除
├── composables/
│   ├── use{Entity}List.ts → 列表逻辑（筛选、分页、排序）
│   └── use{Entity}Form.ts → 表单逻辑（验证、提交）
├── components/
│   ├── {Entity}List.vue   → 列表页
│   ├── {Entity}Detail.vue → 详情页
│   ├── {Entity}Form.vue   → 新增/编辑表单
│   └── {Entity}Card.vue   → 列表项卡片（如果需要）
└── index.ts               → 公共导出

路由配置：
├── /{module}             → 列表页
├── /{module}/create      → 新增页
├── /{module}/:id         → 详情页
└── /{module}/:id/edit    → 编辑页
```

### P3：编码顺序

```
1. types.ts                  → 所有类型定义（先定义清楚数据结构）
2. api/{entity}.ts           → 5 个标准 CRUD 函数
3. composables/use{Entity}List.ts → 列表逻辑
4. composables/use{Entity}Form.ts → 表单逻辑
5. components/{Entity}Card.vue    → 列表项组件
6. components/{Entity}List.vue    → 列表页
7. components/{Entity}Form.vue    → 表单组件
8. components/{Entity}Detail.vue  → 详情页
9. index.ts                       → 公共导出
10. router 注册                   → 路由配置
```

每一步完成后做 P3.2 微自检。

## 修改已有功能流程

### P0：阅读清单（重点！）

```
必读文件（全部完整阅读）：
├── 要修改的文件本身
├── 要修改文件的直接 import 依赖（全部读取）
├── 要修改文件的所有 import 消费者（grep 查找）
└── 相关的类型定义文件

重要：修改已有文件比新建文件风险更高
- 必须理解每一行代码的作用
- 必须知道每一处引用会如何受影响
```

### P2：变更影响分析（重点！）

```
最小变更集：
├── 必须修改的行（列出具体的行号和改动内容）
├── 可能需要调整的调用方（列出文件和调整原因）
└── 确认不需要修改的关联代码（列出排除项）

风险评估：
├── 是否影响公共接口？ → 影响所有调用方
├── 是否影响共享类型？ → 影响所有引用方
├── 是否影响路由？ → 影响导航
└── 是否影响 Store？ → 影响所有消费者
```

### P3：编码原则

```
修改已有代码的额外原则：
1. 先完整阅读要修改的代码段
2. 确认理解每行代码的意图
3. 只修改必要的行，不重新格式化不相关的代码
4. 不改变不相关的代码风格（即使不符合规范）
5. 修改后验证所有调用方不受影响
```

### P4：验证重点

- 修改后的功能正确
- 未修改的功能不受影响
- 调用方无编译错误

## 新增通用组件流程

### P0：阅读清单

```
必读文件：
├── shared/components/ 下所有现有组件    # 确保不重复
├── 最接近的现有通用组件的完整代码       # 参考模式
└── 使用该组件的页面代码（如已知）       # 了解使用场景

重点观察：
├── 现有组件的 props/emits 接口风格
├── 现有组件的 slots 设计
├── 现有组件的 CSS 规范
└── 是否已有功能类似的组件可以扩展
```

### P2：接口设计（重点！）

```
通用组件接口设计检查清单：

Props 设计：
├── 每个 prop 都有必要吗？（不能删除任何一个？）
├── 有合理的默认值吗？
├── 类型够精确吗？（不用 any/过于宽泛的类型）
├── 引用类型默认值用工厂函数了吗？
└── 是否需要支持 v-model？（双向绑定）

Emits 设计：
├── 事件名是否语义明确？
├── 载荷类型是否精确？
├── 是否需要支持 .native 修饰符？
└── 是否与 UI 框架的事件风格一致？

Slots 设计：
├── 默认插槽是否需要作用域数据？
├── 是否需要具名插槽（header/footer/prefix/suffix）？
├── 插槽的 fallback 内容是否合理？
└── 插槽数量是否过多？（>5 个插槽说明组件可能职责过多）

泛型考虑：
├── 是否需要支持不同数据类型？
├── 泛型约束是否合理？
└── 是否与项目中其他泛型组件风格一致？
```

## 重构已有代码流程

### P0：阅读清单（最全面）

```
必读文件：
├── 涉及重构的所有文件（完整阅读）
├── 每个文件的调用方（完整依赖图）
├── 相关的测试文件（如果存在）
└── 相关的类型定义

绘制依赖图：
A → B → C
    ↓
    D → E
确认每个箭头（import 关系）
```

### 重构原则

```
1. 小步前进：每次只做一种重构
2. 每步验证：每步重构后确认功能不变
3. 不混功能：重构中不添加新功能
4. 不改接口：如果必须改接口，先更新所有调用方
5. 可随时停止：每步都是独立的，可以随时停止
```

### 重构顺序

```
1. 提取常量（消除魔法值）
2. 提取类型（统一类型定义）
3. 提取函数（消除重复代码）
4. 提取 composable（分离业务逻辑）
5. 拆分组件（降低单组件复杂度）
6. 调整文件位置（符合架构规范）
```

## 修复 Bug 流程

### P0：问题定位

```
1. 复现条件：什么操作触发了 bug？
2. 预期行为：应该发生什么？
3. 实际行为：实际发生了什么？
4. 影响范围：哪些用户/场景受影响？

阅读重点：
├── 出问题的组件/composable 完整代码
├── 相关的数据流（API → composable → 组件）
├── 相关的状态管理（store/ref）
└── 相关的路由和导航逻辑
```

### 根因分析

```
Bug 定位检查清单：
├── 数据是否正确获取？（检查 API 响应）
├── 数据是否正确转换？（检查 composable 逻辑）
├── 数据是否正确传递？（检查 props 传递）
├── 响应式是否正确？（检查 ref/reactive 使用）
├── 生命周期时序是否正确？（检查 onMounted/watch 时机）
├── 异步操作是否正确处理？（检查 async/await/错误处理）
└── 是否有边界情况未处理？（空值、undefined、类型不匹配）
```

### 修复原则

```
1. 修复根因，不是症状
2. 最小修改，不趁机"优化"
3. 验证修复不引入新问题
4. 考虑是否有类似问题需要一并修复
```

## 紧急变更流程

紧急变更（线上 hotfix）适用 SKILL.md 中定义的**紧急流程降级级别**。

### 简化的流程

```
P0（精简）：只读直接相关的文件，跳过全局分析
P1（跳过）：直接基于已有认知行动（正式降级，非违反铁律）
P2（精简）：只在脑中快速设计，不详细记录
P3（标准）：编码标准不降
P4（加强）：编译验证 + 关键路径手动检查
P5（精简）：只做功能验证，跳过全面审查
```

### 紧急变更不降低的标准

- P3 编码标准不降（不引入 any、不跳过类型定义）
- P4 编译验证不跳过（至少做类型检查）
- 最小改动原则不放松
- 交付输出必须标注"执行了紧急流程降级"及原因

### 紧急变更后补做的事项

```
□ 补做完整的 P0-P5 流程
□ 检查是否有架构层面的改进空间
□ 补充缺失的类型定义或错误处理
□ 补充测试用例
```

## 全新项目初始化流程

当目标目录为空或用户明确要求"从零搭建项目"时，不走标准七阶段流水线，
而是走以下初始化流程。初始化完成后，后续的功能开发走标准流水线。

### 第一步：需求确认（必须向用户确认）

```
【项目初始化确认】
1. 项目名称：
2. 项目类型：SPA / SSR (Nuxt) / Library
3. UI 框架偏好：Element Plus / Ant Design Vue / Naive UI / 无
4. CSS 方案偏好：Tailwind / Scoped CSS / CSS Modules
5. 是否需要状态管理：是 (Pinia) / 否
6. 是否需要路由：是 (vue-router) / 否
7. 包管理器：pnpm / npm / yarn
8. 其他特殊需求：（如 i18n、微前端、移动端适配等）

请确认或补充。
```

### 第二步：项目脚手架

使用官方或社区成熟的脚手架工具，禁止手动一个个文件创建：

```bash
# SPA 项目
pnpm create vite {project-name} --template vue-ts

# Nuxt 项目
npx nuxi@latest init {project-name}

# 或使用 create-vue（官方推荐，支持 router/pinia/eslint 预配置）
pnpm create vue@latest
```

### 第三步：安装核心依赖

```bash
cd {project-name}

# 路由（如脚手架未包含）
pnpm add vue-router

# 状态管理（如需要）
pnpm add pinia

# UI 框架（根据用户选择）
pnpm add element-plus   # 或 ant-design-vue / naive-ui

# HTTP 客户端
pnpm add axios

# 开发依赖
pnpm add -D @types/node vitest @vue/test-utils jsdom
```

### 第四步：建立项目骨架

根据 `references/architecture.md` 的规模自适应规则，建立目录结构：

```
判断项目规模（默认按"小型"开始）：
├── 页面 ≤5 个 → 小型架构
├── 页面 5-20 个 → 中型架构（features/ 结构）
└── 页面 20+ → 大型架构（modules/ 结构）

宁可低一级也不要过度设计，后续可以升级。
```

必做的骨架文件：

```
必须创建：
├── src/styles/variables.css    → CSS 变量定义（参考 code-standards.md）
├── src/styles/global.css       → 全局样式重置
├── src/utils/http.ts           → HTTP 实例和拦截器（参考 architecture.md）
├── src/types/global.d.ts       → 全局类型声明
├── src/router/index.ts         → 路由配置（如需要）
├── src/App.vue                 → 根组件
└── src/main.ts                 → 入口文件

按需创建：
├── src/stores/                 → Pinia store（如需要状态管理）
├── src/shared/components/      → 通用组件目录
├── src/composables/            → 通用 composables
├── src/layouts/                → 布局组件（如需要多布局）
└── vitest.config.ts            → 测试配置
```

### 第五步：验证初始化

```bash
# 安装依赖
pnpm install

# 类型检查
npx vue-tsc --noEmit

# 构建验证
pnpm run build

# 启动开发服务器
pnpm run dev
```

### 第六步：交付

向用户交付初始化结果：

```
【项目初始化完成】
- 项目名称：{name}
- 技术栈：Vue 3 + TypeScript + Vite + {UI 框架} + {CSS 方案}
- 目录结构：{小型/中型/大型} 架构
- 已安装的核心依赖：（列出）
- 已创建的骨架文件：（列出）
- 开发服务器：已验证可正常启动

【后续建议】
- 下一步可以开始创建第一个页面，将走标准流水线
- 建议先配置好 .env 文件中的 API 地址
```

**重要**：项目初始化完成后，后续所有功能开发（新增页面、组件等）必须走标准七阶段流水线。

## Git 工作流

### 核心原则

- **每次改动可追溯**：所有变更通过 git 管理，禁止"改完就走"
- **安全可回退**：每个阶段完成后的状态都可以回退到
- **不做超出范围的 git 操作**：不主动 push、不主动创建远程分支、不修改 git 配置

### 分支策略

收到任务后，根据项目当前的 git 状态决定：

```
当前分支情况？
├── 用户已在功能分支上 → 在当前分支工作
├── 用户在 main/master 上
│   ├── 任务复杂度 ≥ 标准 → 提醒用户"建议创建功能分支"，等用户决定
│   └── 任务复杂度 = 简单 → 在当前分支工作（改动小，直接提交即可）
└── 项目没有 git 仓库 → 提醒用户初始化 git，不主动执行 git init
```

**禁止**：不主动创建分支、不主动切换分支、不主动 push，除非用户明确要求。

### 提交策略

| 时机 | 提交内容 | commit message 格式 |
|------|---------|-------------------|
| P2 用户确认后 | 当前状态（如有未提交的改动） | 不提交，等编码完成 |
| P3 编码完成后 | 所有新增/修改的文件 | `feat: {简述做了什么}` |
| P4 修复循环中 | 每轮修复后的文件 | `fix: {修复的问题}` |
| P5 回验补充后 | 补充的文件 | `feat: 补充 {缺失的功能}` |
| P6 最终审查后 | 无新提交（如有微调则提交） | `chore: {微调内容}` |

**实际操作**：AI 不主动执行 git commit，但在交付输出中提供推荐的 commit message，
由用户决定是否提交。如果用户要求 AI 直接提交，则按上述格式执行。

### 变更保护

在编码开始前：

```bash
# 记录当前状态，确保可以回退
git stash list    # 检查是否有未保存的改动
git status        # 确认工作区状态
git diff          # 查看具体变更内容
```

如果发现工作区有未提交的改动：
- 提醒用户"当前有未提交的改动，建议先 stash 或 commit"
- 不主动 stash 用户的改动
- 等用户处理后再继续

### 分支操作命令

```bash
# 查看当前分支
git branch        # 列出本地分支，* 表示当前分支
git branch -vv    # 查看分支详细信息（含上游分支）

# 创建功能分支（仅在用户明确要求时执行）
git checkout -b feat/{功能名称}     # 创建并切换到新分支
git checkout -b fix/{问题描述}     # 创建修复分支

# 切换分支（仅在用户明确要求时执行）
git checkout {分支名}              # 切换到已有分支
```

### 提交命令

```bash
# 查看将要提交的内容
git diff --cached    # 查看已暂存的变更（git add 后）
git diff            # 查看未暂存的变更

# 暂存文件
git add {文件路径}      # 暂存单个文件
git add src/views/     # 暂存整个目录
git add -A            # 暂存所有变更（包含新增、修改、删除）

# 提交变更
git commit -m "feat: {简述做了什么}"           # 提交并写 message
git commit -m "fix: {修复的问题}"               # 修复类提交
git commit -m "chore: {微调内容}"               # 杂项提交

# 提交时查看历史记录（了解提交风格）
git log --oneline -10   # 查看最近10条提交
```

### 交付时的 Git 信息

P6 交付输出中包含：

```
【Git 变更说明】
- 分支：{当前分支名}
- 状态：{是否有未提交的改动}
- 推荐的 commit message：
  feat: {功能描述}

  - 新增 {文件1}：{用途}
  - 修改 {文件2}：{修改内容}
  - ...
```
